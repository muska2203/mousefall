# PROMPT_GUIDE — Как писать промпты в новом подходе

> Это руководство для пользователя, не для агента. Оно описывает, как формулировать задачи после перехода на protocol-driven workflow.

---

## Главный принцип

Промпт должен быть **кратким**. Агент сам прочитает `AGENTS.md`, `PROTOCOL.md` и `SYNC_STATUS.md` и определит, что делать.

Не нужно больше писать:
- «прочитай документацию»;
- «сначала спроси меня, если непонятно»;
- «не забудь тесты»;
- «действуй по минимуму».

Всё это уже в протоколах.

---

## Базовый шаблон промпта

```
[Краткое описание цели]

[Контекст, если нужен]

[Ограничения или вопросы]
```

---

## Примеры промптов по типам задач

### Добавление контента

```
Добавь врага "rat_scout".
- 10 HP, 2 AP
- стратегия hunter
- оружие common_venom_dagger
- дроп: health_potion с небольшим шансом
```

Агент сам:
- выберет `content-addition` протокол;
- откроет `docs/recipes/add-enemy.md`;
- возьмёт шаблон `public/content/examples/enemy-template.json`;
- добавит тексты;
- зарегистрирует в манифесте;
- запустит проверки.

---

```
Добавь оружие "rusty_dagger".
- 4 урона, ближний бой
- теги: attack.melee, delivery.weapon, damage.physical.slashing
- пассивка: 20% шанс наложить bleeding на 2 хода
```

Агент сам поймёт, что нужно добавить и оружие, и контентное правило.

---

### Системная механика

```
Сделай так, чтобы оглушённый актор получал двойной урон от ближних атак.
```

Агент:
- выберет `system-design` протокол;
- проверит `SYNC_STATUS.md`;
- предложит план через `ContentRule` или модификатор урона;
- при 3+ системах включит двухфазный режим.

---

```
Добавь новый интент KNOCKBACK, который отталкивает цель на 1 клетку.
```

Агент откроет `docs/recipes/add-intent.md`.

---

### Исправление бага

```
Исправь баг: враг иногда проходит сквозь закрытые двери.
```

Агент:
- выберет `bug-fix` протокол;
- найдёт или напишет тест;
- локализует причину;
- предложит минимальное исправление.

---

### Рефакторинг

```
Вынеси логику расчёта урона из executeDamageIntent в отдельную чистую функцию.
```

Агент:
- выберет `refactor` протокол;
- найдёт тесты;
- сделает минимальное изменение;
- проверит, что тесты проходят.

---

### Разбор «как работает»

```
Разберись, как работает порядок применения контентных правил при ENTITY_DAMAGED.
```

Агент:
- выберет `investigation` протокол;
- прочитает `docs/agents/CONTENT_RULES_EDGE_CASES.md`;
- посмотрит код `src/simulation/content-rules/reaction/`;
- даст краткий ответ без изменений.

---

### Актуализация документации по долгу из SYNC_STATUS.md

```
Актуализируй документацию по первому пункту из раздела "Известный рассинхрон" в docs/agents/SYNC_STATUS.md.

Источник правды — код и тесты. После исправления обнови или удали запись в SYNC_STATUS.md.
```

Агент:
- выберет `docs_update` протокол;
- прочитает `SYNC_STATUS.md`;
- найдёт указанный документ и соответствующий код;
- приведёт документацию в соответствие с кодом;
- обновит `SYNC_STATUS.md`.

Если рассинхрон серьёзный и требует обсуждения архитектуры — агент остановится и спросит.

## Что добавлять в промпт, а что нет

| Не нужно | Почему |
|---|---|
| «Прочитай документацию» | Агент делает это автоматически по `AGENTS.md`. |
| «Не меняй лишнее» | В `PROTOCOL.md` есть правило минимальных изменений. |
| «Запусти тесты» | Это в post-task чеклисте `AGENTS.md`. |
| «Спроси, если непонятно» | Это в escalation rules `PROTOCOL.md`. |
| Длинный контекст задачи | Если фон важен — укажи ссылку на документ или файл. |

| Полезно добавить | Когда |
|---|---|
| Конкретные ID | «враг cat_small», «оружие common_flaming_sword». |
| Числа и параметры | HP, урон, шанс, длительность. |
| Ограничения | «без изменения AI», «только для ближнего боя». |
| Вопросы | Если хочешь обсудить варианты до реализации. |

## Когда нужен двухфазный режим

Если задача затрагивает несколько систем, агент сам предложит сначала план. Но можно указать это явно:

```
Хочу добавить новый тип урона "electric". Сначала предложи план, не пиши код.
```

---

## Что делать, если агент ошибся с типом задачи

Если агент выбрал не тот протокол, напиши:

```
Это не content_addition, это system_design. Перечитай PROTOCOL.md и предложи план.
```

После нескольких таких случаев стоит уточнить `PROTOCOL.md` — добавить ключевые слова или примеры.

---

## Поддержка качества

Если после задачи документация устарела — не правь руками, а дай агенту:

```
Актуализируй docs/agents/ACTION_SYSTEM.md под текущий код после последних изменений.
```

Агент выберет `docs_update` протокол и сделает это сам.

---

## Страховочный префикс (если агент не пошёл по протоколу)

Иногда агент может пропустить начальное чтение `AGENTS.md` / `PROTOCOL.md` — например, если системный контекст не подгрузил нужные файлы. В таких случаях вставьте этот префикс первым сообщением в сессии:

```text
Перед выполнением задачи строго следуй воркфлоу из AGENTS.md:
1. Прочитай корневой AGENTS.md и docs/agents/PROTOCOL.md.
2. Определи тип задачи по PROTOCOL.md и следуй соответствующему протоколу из docs/agents/protocols/.
3. Если работаешь в подпапке src/X/, прочитай её локальный AGENTS.md.
4. Прочитай docs/agents/SYNC_STATUS.md, если задача касается документации или [DRAFT]-систем.
5. В конце: запусти тесты изменённой области, npm run typecheck и обнови документацию/SYNC_STATUS.md при необходимости.
```

После этого можно задавать задачу обычным кратким промптом.

---

## Итог

После перехода твои промпты должны выглядеть так:

```
Добавь врага X с параметрами Y.
```

```
Сделай так, чтобы Z.
```

```
Исправь баг: ...
```

Вся остальная работа по навигации, протоколам, тестам и документации — на агенте.
