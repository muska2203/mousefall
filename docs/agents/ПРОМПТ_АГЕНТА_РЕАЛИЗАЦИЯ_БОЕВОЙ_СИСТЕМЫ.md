# Промпт агента: реализация боевой системы на основе реакций

> Этот файл — инструкция для агента, который будет помогать внедрять
> проработанный концепт боевой системы в существующую кодовую базу.
>
> **Важно:** концепт уже зафиксирован, следующий шаг — плавная реализация
> с минимальным риском для рабочего кода.

---

## 1. Цель

Внедрить в игру `Mousefall` боевую систему, построенную вокруг
`ContentRuleReaction`, `RuleContext`, `activeRules` и двух фаз выполнения
правил (модификаторы на интенте + реакции на событии).

Существующий код уже реализует похожие идеи через `WorldReaction` в
`src/simulation/systems/world-reactions/`. Нужно не сломать текущую игру,
а постепенно заменить / дополнить старые механизмы новыми.

## 3. Рекомендуемый процесс: постепенная миграция (strangler fig)

У нас есть рабочий код, который делает боевую логику по-своему. Нельзя
переписать всё сразу — риск сломать игру слишком велик.

Рекомендуется следующий подход:

1. **Сначала добавить новый механизм рядом со старым.**
2. **Проверить, что старый механизм продолжает работать.**
3. **Перенести одну маленькую фичу из старого механизма в новый.**
4. **Проверить через тесты и, если возможно, через игру.**
5. **Повторять, пока старый механизм не станет ненужным.**
6. **Удалить старый механизм только после полной миграции.**

Это классический **strangler fig** — новая система постепенно "обвивает"
старую, заменяя её по частям.

## 4. Общая последовательность этапов

### Этап 0. Аудит и подготовка

**Цель:** понять, что уже есть в коде и как это соотносится с концептом.

- Изучить все файлы из раздела 2.
- Составить таблицу соответствия:
  - старая `WorldReaction` → новое контентное правило (или системная реакция);
  - старый `Intent` → нужен ли он в новой системе;
  - текущие события → как они ложатся в `RuleContext`.
- Зафиксировать ограничения текущего кода, которые могут повлиять на
  реализацию (например, отсутствие `sourceEntityId` у некоторых интентов).
- **Не писать код.** Только исследование и документирование.

### Этап 1. Скелет `ContentRuleReaction`

**Цель:** создать новый модуль, который пока ничего не ломает.

- Создать модуль `src/simulation/systems/content-rules/`.
- Определить типы:
  - `ContentRule`;
  - `OwnerContext`;
  - `RuleContext`;
  - `ActiveRule`.
- Реализовать `buildRuleContext(event, state)` — функцию, которая строит
  `RuleContext` для событий/интентов.
- Реализовать `runContentRuleReaction(state, ruleContext)` — пока заглушку,
  возвращающую пустой массив `Intent[]`.
- Добавить unit-тесты на `buildRuleContext`.

### Этап 2. Интеграция в игровой цикл

**Цель:** подключить `ContentRuleReaction` к существующему потоку без
изменения логики.

- В `executeIntent` (или аналогичном месте) добавить фазу модификаторов
  **перед** вызовом `IntentExecutor`.
- После `IntentExecutor` добавить вызов `runContentRuleReaction` **перед**
  `runWorldReactions`.
- Системные `WorldReaction` остаются на месте.
- Убедиться, что все существующие тесты проходят.

### Этап 3. Первые контентные правила

**Цель:** проверить, что новый механизм работает на простом примере.

- Выбрать одну-две существующие мировые реакции, которые описывают физику
  (например, `collision_daze` или `burning_tick_damage`).
- Перевести их в декларативные `ContentRule`.
- Добавить эти правила в глобальный реестр мировых контентных правил.
- Убедиться, что старая реакция и новое правило не дублируют эффект.
- Добавить unit-тесты.

### Этап 4. `activeRules` у актора

**Цель:** реализовать кэш активных правил.

- Добавить поле `activeRules` к актору (или к `GameState`, если актора пока
  нет в нужном виде).
- Реализовать добавление/удаление правил при:
  - `EQUIP_ITEM` / `UNEQUIP_ITEM`;
  - `STATUS_APPLIED` / `STATUS_REMOVED`;
  - `GRANT_ABILITY` / `REVOKE_ABILITY`.
- Правильно заполнять `ownerContext`:
  - `itemInstanceId` для экипировки;
  - `statusInstanceId` для статусов;
  - `abilityId + ':' + sourceItemInstanceId` для талантов.
- В `ContentRuleReaction` собирать правила из `activeRules` для слоёв
  `source`, `target` и `radius`.
- Добавить тесты на добавление/удаление правил.

### Этап 5. Перевод оставшихся мировых реакций

**Цель:** перенести базовую физику из кода в data-driven правила.

- По одному переводить оставшиеся физические реакции в контентные правила:
  - `collision_damage`;
  - `burning_tick_damage`;
  - правила тайловых эффектов (`water_applies_wet`, `oil_applies_oiled` и т.д.).
- После каждого перевода удалять старую кодовую реакцию.
- Поддерживать прохождение тестов.

### Этап 6. Модификаторы на интенте

**Цель:** реализовать фазу модификаторов.

- Реализовать эффект `MODIFY_DAMAGE`.
- Реализовать сортировку модификаторов:
  - слои `source → target → world → radius`;
  - внутри слоя `multiply` → `add`;
  - `priority` → `ruleId`.
- Реализовать `ParametrizedValue` для числовых полей.
- Перевести правила оружия в контентные правила:
  - `piercing_crit`;
  - `blunt_melee_stun`;
  - `slashing_causes_bleeding`;
  - `blunt_push_wall_stun`.

### Этап 7. Скиллы оружия и шаблоны предметов

**Цель:** убедиться, что скиллы оружия остаются простыми генераторами
интентов, а вся логика живёт в правилах.

- Проверить, что скиллы создают простые интенты с правильными тегами.
- Добавить `ruleIds` в шаблоны оружия, если их ещё нет.
- Добавить `ruleIds` в шаблоны колец, амулетов и других предметов.

### Этап 8. Окружение и стихийная система

**Цель:** реализовать сбор правил окружения.

- Реализовать сбор правил из `tileEffects` по `eventPosition`.
- Реализовать сбор правил из шаблонов тайлов.
- Перевести стихийные комбо (`wet` / `oiled` / `burning` + `water` / `oil` /
  `fog`) в контентные правила.
- Проверить цепочки с огнём, маслом и водой.

### Этап 9. Инструменты валидации и отладки

**Цель:** сделать систему поддерживаемой.

- Добавить валидацию `ruleIds` в шаблонах предметов/статусов/талантов.
- Добавить dev-дамп активных правил актора для отладки.
- Добавить логирование цепочки реакций (если ещё не реализовано).

### Этап 10. Финальная чистка

**Цель:** удалить остатки старой системы.

- Удалить устаревшие кодовые реакции, если они полностью заменены.
- Унифицировать имена модулей и функций.
- Обновить документацию (`ACTION_SYSTEM.md`, `EVENT_FLOW.md`, при необходимости
  `AGENTS.md`).

## 5. Текущий фокус: с чего начать прямо сейчас

Если агент получает этот промпт впервые, его **первая задача** — выполнить
**Этап 0 и Этап 1**:

1. Проанализировать текущий код и составить карту соответствия
   "старые реакции → новые правила".
2. Создать скелет `ContentRuleReaction` с типами и заглушкой.
3. Добавить unit-тесты на `buildRuleContext`.
4. Написать краткий отчёт:
   - что найдено в коде;
   - какие ограничения есть;
   - какие риски видны;
   - что реализовано.

## 6. Правила работы для агента

### Что делать

- Работать **поэтапно**. Один PR / один коммит — один этап.
- После каждого изменения запускать существующие тесты:
  ```bash
  npm test
  ```
  или аналогичную команду из `package.json`.
- Добавлять новые unit-тесты на каждый добавленный механизм.
- Если что-то ломается — откатить изменения и обсудить с пользователем.
- Следовать стилю существующего кода.
- Все комментарии в коде — на русском (имена переменных/типов/функций — на
  английском).

### Чего не делать

- **Не переписывать всё сразу.** Только поэтапная миграция.
- **Не удалять старые реакции**, пока не убедишься, что новые полностью их
  заменяют.
- **Не менять логику скиллов**, если это не требуется для интеграции.
- **Не добавлять составные правила** (одно правило → несколько разных
  интентов) до финала миграции.
- **Не углубляться в баланс** контента — фокус на архитектуре и
  инфраструктуре.

### Как принимать решения

- Если требования концепта противоречат существующему коду — задавать
  уточняющие вопросы пользователю.
- Если есть несколько вариантов реализации — предлагать их с краткой
  аргументацией за/против.
- Не вводить новые зависимости без согласования.
