# Протокол: system_design

## Когда применять

Добавление или изменение игровой механики, системных компонентов, типов интентов, реакций, AI, сохранений.

---

## Последовательность действий

### 1. Контекст

- Прочитай [`SYNC_STATUS.md`](../SYNC_STATUS.md).
- Прочитай разделы документации, связанные с затрагиваемой системой.
- Если система `[STABLE]` — доверяй документации.
- Если система `[DRAFT]`/`[EXPERIMENTAL]` — читай код как источник правды.

### 2. Оценка масштаба

- **Локальная задача** (1–2 системы): кратко опиши план, задай вопросы, дождись ответа, реализуй.
- **Кросс-системная** (3+ системы): включи двухфазный режим.

### 3. Двухфазный режим

**Фаза 1 — Проектирование:**
- Опиши предполагаемые изменения списком (3–7 пунктов).
- Укажи, какие файлы документации нужно обновить.
- Укажи, какие тесты добавить/изменить.
- Задай уточняющие вопросы.
- Не трогай код.

**Фаза 2 — Реализация** (только после явного утверждения):
- Реализуй по утверждённому плану.
- Обнови/добавь тесты.
- Обнови документацию, если система стабильна или стала стабильнее.
- Запусти тесты.

### 4. Финал

- Сообщи, что изменено.
- Укажи статус тестов.
- Отметь, если нужно обновить [`SYNC_STATUS.md`](../SYNC_STATUS.md).
