# TRANSITION_PLAN — Переход на protocol-driven работу с агентом

> Этот файл описывает план полного перехода проекта на новый подход: агент сам определяет тип задачи, читает соответствующий протокол и действует по нему.

---

## Цель

Сделать так, чтобы для любой рутинной задачи пользователю не приходилось писать длинный промпт. Достаточно краткой формулировки цели — агент сам выбирает протокол, проверяет статус документации и действует предсказуемо.

## Фазы перехода

### Фаза 1. Минимальный рабочий набор рецептов

**Статус:** выполняется в текущей сессии.

Создать рецепты, без которых `content-addition` протокол бесполезен:

- [x] `docs/recipes/add-enemy.md`
- [x] `docs/recipes/add-weapon.md`
- [x] `docs/recipes/add-status.md`
- [x] `docs/recipes/add-content-rule.md`

Критерий готовности: по запросу «добавь врага X» агент может открыть рецепт и выполнить задачу без дополнительных промптов.

### Фаза 2. Расширенный набор рецептов

Добавить рецепты для остальных типов контента:

- `docs/recipes/add-armor.md`
- `docs/recipes/add-amulet.md`
- `docs/recipes/add-consumable.md`
- `docs/recipes/add-ability.md`
- `docs/recipes/add-tile-effect.md`
- `docs/recipes/add-map.md`

### Фаза 3. Аудит существующей документации

**Источник правды:** код + тесты.

Проверить каждый `[STABLE]`-документ на актуальность:

1. Прочитать документ.
2. Проверить все пути к файлам кода.
3. Проверить, что примеры кода соответствуют текущей реализации.
4. Проверить, что чеклисты актуальны.
5. При обнаружении рассинхрона — либо исправить документ, либо перевести в `[DRAFT]` и занести в `SYNC_STATUS.md`.

Приоритет аудита:
1. `ACTION_SYSTEM.md`
2. `CONTENT.md`
3. `TURN_FLOW.md`
4. `PRESENTATION_CONTRACT.md`
5. `TESTING.md`
6. `CONTENT_RULES_EDGE_CASES.md`
7. `TILE_EFFECTS.md`
8. `I18N.md`
9. `LAYERS.md`
10. `GLOSSARY.md`

### Фаза 4. Рефакторинг монолитных справочников

Разделить большие документы на:
- короткий справочник с принципами (`[STABLE]`);
- рецепты для конкретных действий.

Цели:
- `ACTION_SYSTEM.md` → `ACTION_SYSTEM.md` (короткий) + рецепты `add-action.md`, `add-event.md`.
- `TURN_FLOW.md` → оставить как справочник, возможно сократить.
- `AI_SYSTEM.md` → когда система устаканится, превратить в справочник + рецепт `add-ai-strategy.md`.
- `CONTENT_RULES_EDGE_CASES.md` → оставить как справочник, добавить рецепт `add-content-rule.md`.

### Фаза 5. Живые примеры контента

Создать `public/content/examples/` с образцовыми JSON-файлами:

### Фаза 6. Валидация нового подхода

Провести 1–2 реальных задачи строго по новому подходу:

1. Пользователь даёт краткую формулировку.
2. Агент читает `PROTOCOL.md` и `SYNC_STATUS.md`.
3. Агент выбирает протокол.
4. Агент следует рецепту.
5. Пользователь оценивает результат.

По итогам скорректировать протоколы и рецепты.

### Фаза 7. Поддержка в актуальном состоянии

- После каждой задачи, затрагивающей нестабильную систему, обновлять `SYNC_STATUS.md`.
- Раз в несколько задач проверять `SYNC_STATUS.md` на устаревшие записи.
- При стабилизации системы переводить документ из `[DRAFT]` в `[STABLE]` и актуализировать.
