## 2. Ключевые решения, к которым пришли

### 2.1. Скиллы остаются как сейчас

- У каждого оружия должен быть минимум три скилла:
  - **атакующий** (обычно `1`);
  - **вспомогательный** (обычно `2`);
  - **защитный** (обычно `3`).
- Раскладка `1–3` — ориентировочная, не жёсткая.
- Скиллы должны быть **тупыми генераторами интентов**:
  - проверяют дальность/цель/кулдаун/ОД;
  - создают простые интенты (`DAMAGE`, `PUSH`, `APPLY_STATUS` и т.д.);
  - проставляют **все нужные теги**;
  - не реализуют комбо и не лезут в пассивки.

### 2.2. Два слоя мировых реакций

1. **Системные мировые реакции** — код, инварианты мира:
   - смерть при `hp <= 0`;
   - дроп лута;
   - уведомление AI;
   - очистка трупов и переходы этажей.

   Базовые физические эффекты (например, урон и `dazed` от столкновений) вынесены в **мировые контентные правила**.

2. **Контентные правила через `ContentRuleReaction`** — data-driven правила от предметов, талантов, статусов, окружения и мира.

> **Примечание:** динамические мировые правила (например, временные ауры уровня или глобальные модификаторы сложности) могут появиться позже, но будут прорабатываться отдельно и, скорее всего, реализованы через контентные правила или отдельный механизм, а не через системный реестр реакций.

### 2.3. Две фазы выполнения правил

| Фаза | Когда | Для чего |
|---|---|---|
| **Модификаторы на интенте** | Перед `IntentExecutor` | Изменить урон, дальность, длительность статуса и т.д. |
| **Реакции на событии** | После `IntentExecutor` | Наложить статус, создать дополнительный урон, дать ОД и т.д. |

Порядок модификаторов: сначала по слою `source → target → world → radius`, внутри слоя сначала `multiply`, затем `add`, затем по `priority`, при равных `priority` — по `ruleId`.

Порядок реакций: сначала `ContentRuleReaction` (контентные реакции, включая мировые правила), затем системные мировые реакции (только инварианты). Интенты, порождённые контентными реакциями, выполняются рекурсивно до системных реакций на исходное событие.

### 2.4. Формат и хранение правила

Каждое правило — отдельный data-driven объект в TypeScript. Шаблоны предметов, статусов и талантов (сами шаблоны остаются JSON) ссылаются на правила по `ruleIds`.

```text
rule:
  id: slashing_causes_bleeding
  ownerContext:
    type: entity
    entityId: abc-123
  trigger:
    event: ENTITY_DAMAGED
    tags: [damage.physical.slashing]
  targetConditions:
    - type: chance
      value: 0.3
  effect:
    type: APPLY_STATUS
    status: bleeding
    duration: 3
  target: eventTarget
  priority: 0
```

Основные поля:

- `id` — уникальный идентификатор правила.
- `ownerContext` — владелец правила в рантайме (заполняется при добавлении в `activeRules`).
- `trigger.event` — событие или интент (`ENTITY_DAMAGED`, `DAMAGE_INTENT`, `ENTITY_HEALED` и т.д.).
- `trigger.tags` — обязательные теги.
- `conditions` — глобальные декларативные операторы, проверяемые один раз: `chance`, `hasStatus`, `statCompare`, `distance`, `faction`, `inTileEffect`, `and`, `or`, `not`.
- `targetConditions` — декларативные операторы, проверяемые отдельно для каждой цели, найденной селектором `target`.
- `effect` — что делает правило.
- `target` — селектор целей (`eventTarget`, `eventSource`, `self`, `collisionTarget`, `allInRadius`, `nearestEnemy`, `chain`).
- `priority` — порядок срабатывания.

### 2.5. Откуда собирать правила

Правила собираются из четырёх источников:

1. **Источник действия** — кто совершил действие.
2. **Цель действия** — на ком оно отразилось.
3. **Мир / окружение** — глобальные мировые правила (физика столкновений), тайлы, лужи, туман, горящие клетки.
4. **Сущности в радиусе события** — для аур и наблюдателей.

Для сбора правил `ContentRuleReaction` использует **RuleContext** — контекст
события с полями `sourceEntityId`, `targetEntityId`, `eventPosition`,
`eventTags` и специфичными полями (`collisionTargetId`, `abilityTargetPosition`).
Подробнее — в концепте, раздел 2.2.3.

Разрешение `self` зависит от слоя источника:

| Слой | `self` разрешается в |
|---|---|
| `source` | Атакующий (обычно `eventSource`). |
| `target` | Цель действия (обычно `eventTarget`). |
| `radius` | Сторонний наблюдатель / источник ауры. |
| `world` | Не используется; правила мира работают с `eventSource`, `eventTarget`, `eventPosition`. |

Пример наблюдателя:

```text
trigger:
  event: ENTITY_HEALED
conditions:
  - type: distance
    from: self
    to: eventTarget
    max: 3
  - type: faction
    target: eventTarget
    relation: enemy
    to: self
effect:
  type: APPLY_STATUS
  status: rage
  duration: 2
target: self
```

> **Примечание:** здесь `self` — это владелец правила (наблюдатель), а не источник или цель события. Правило срабатывает, когда враг лечится в радиусе 3 клеток от наблюдателя, и накладывает `rage` на самого наблюдателя.

### 2.6. Жизненный цикл правил

> ✅ **Решено.** Подробное описание — в
> [`docs/plans/Жизненный_цикл_правил_и_activeRules.md`](../plans/Жизненный_цикл_правил_и_activeRules.md).

- Шаблоны предметов, статусов и талантов содержат `ruleIds`; сами правила хранятся в TypeScript-модулях как декларативные объекты.
- **Кэшируются активные правила у актора** (`actor.activeRules`). Это производный кэш: первичными источниками остаются экипировка, статусы и таланты.
- При надевании/снятии предмета, наложении/снятии статуса, получении/отзыве таланта правила добавляются/удаляются.
- Каждый `ActiveRule` помечается источником через `ownerContext`:
  - экипировка — по `itemInstanceId`;
  - статусы — по `statusInstanceId` (уникальному для каждого экземпляра статуса);
  - таланты — по `abilityId + ':' + sourceItemInstanceId` (чтобы различать один и тот же талант из разных источников).
- `self` внутри правила разрешается строго через `ownerContext`, а не через `eventSource` / `eventTarget`.
- Правила окружения не кэшируются у актора, а определяются по позиции события.
- `activeRules` не сохраняется в save-файл; он пересобирается при загрузке из экипировки, статусов и талантов.
- Изменения `activeRules` применяются немедленно, но текущая фаза `ContentRuleReaction` работает со снимком правил, собранным в начале фазы.

### 2.7. Один интент на правило

> ✅ **Решено:** одно правило описывает ровно один шаблон намерения.

- Каждое правило имеет ровно один `effect`, который описывает один шаблон `Intent`.
- Если источнику (предмет, статус, талант) нужно несколько разных эффектов, в его шаблоне перечисляется несколько `ruleIds`.
- Порядок выполнения таких эффектов определяется `priority` и `ruleId` внутри фазы.
- Условия правила (`chance`, `hasStatus` и т.д.) проверяются отдельно для каждого правила.
- При использовании мультитаргетного селектора (`allInRadius`, `chain`) один шаблон выполняется по одному разу для каждой подходящей цели, порождая несколько конкретных инстансов `Intent`.
- Составные правила, порождающие несколько разных типов интентов за одно срабатывание, сознательно отложены. Если они понадобятся, для них будет введён отдельный механизм.

### 2.8. Ограничения

- Вероятности должны использовать детерминированный рандом.
- Ограничение на глубину цепочки реакций — например, **1000 реакций за один шаг**.
- Модификаторы должны быть чистыми: меняют только значения в интенте, не мутируют состояние мира.
- Контентные реакции выполняются перед системными мировыми реакциями.
- Порядок правил: `source → target → world → radius`, затем `priority`, затем `ruleId`.
- Внутри слоя `world`: глобальные мировые правила → тайловые эффекты → статические свойства тайла.
- Для модификаторов `multiply` / `add` группируются **внутри каждого слоя**, а не глобально.
- Для реакций все `REMOVE_STATUS` выполняются перед всеми `APPLY_STATUS`; после всех `APPLY_STATUS` централизованно разрешаются конфликты через `blockedBy` / `statusCategory` / `categoryPriority`.
- Self-эффекты (`sourceEntityId === targetEntityId`) не дублируют правила: слои `source` и `target` для одной сущности объединяются.
- Правило срабатывает один раз на объект события, но может срабатывать многократно в рамках одной цепочки, если происходит несколько подходящих событий.
- Изменения `activeRules` применяются немедленно, но текущая фаза `ContentRuleReaction` работает со снимком правил, собранным в начале фазы.

### 2.9. Тайловые эффекты и окружение

> **Статус реализации:** вынесено из MVP. Запланировано в фазе 7 «Тайловые эффекты и окружение» после окончательного перехода на новый концепт.

- Динамические эффекты окружения хранятся в отдельном слое `tileEffects` в `GameState`.
- В одной клетке может быть несколько эффектов разных типов, но только один эффект каждого типа; новый экземпляр обновляет длительность/интенсивность.
- Тайловые эффекты тикают в `round-recovery` / на `BEGIN_TURN` нового раунда.
- Правила окружения хранятся в TypeScript-объектах тайловых эффектов и шаблонах тайлов; собираются `ContentRuleReaction` по позиции события.
- Базовые физические переходы (огненный урон + масло → горящее масло + взрыв; вода + горящее масло → масло потушено + пар) выполняются напрямую и порождают событие `TILE_EFFECT_CHANGED`.
- Огненный урон поджигает `oil` в клетке и в радиусе 1; горящее масло распространяется на соседнее масло.
- Стартовый набор эффектов: `water`, `oil` (может быть `ignited`), `fog`.
- Сложные зональные эффекты (взрыв горящего масла) представлены интентом `EXPLOSION`, который порождает событие `EXPLOSION_TRIGGERED`; реакции на него наносят урон и поджигают соседнее масло.

## 3. Открытые вопросы и проблемы

Ниже список вопросов, которые ещё не проработаны. Их нужно обсуждать и решать.

### 3.1. Формат хранения правил

> ✅ Решено.

- Правила хранятся как декларативные объекты в TypeScript-модулях.
- Шаблоны предметов, статусов и талантов ссылаются на правила по `ruleIds`.
- Для условий используется декларативный формат; отдельный DSL откладывается.

### 3.2. Типы эффектов

> ✅ Core-набор решён.

Минимальный набор для старта:

| Эффект | Фаза | Описание | Параметризуемые поля |
|---|---|---|---|
| `MODIFY_DAMAGE` | Модификаторы на интенте | Изменяет урон интента и может добавлять теги | `add`, `multiply` (константа или `ParametrizedValue`), `addTags` |
| `APPLY_STATUS` | Реакции на событии | Накладывает статус на цель | `duration`, `stacks` |
| `REMOVE_STATUS` | Реакции на событии | Снимает один или несколько статусов с цели | `statusTypes`, `allStacks?` |
| `DEAL_DAMAGE` | Реакции на событии | Наносит дополнительный урон | `amount` |
| `HEAL` | Реакции на событии | Восстанавливает здоровье | `amount` |
| `RESTORE_AP` | Реакции на событии | Восстанавливает очки действия | — |
| `CONSUME_AP` | Реакции на событии | Тратит очки действия у владельца правила | `amount` |

Числовые поля эффектов задаются через `ValueSpec`:

- `number` — константа;
- `ParametrizedValue` — объект `{ ref, multiply?, divide?, add?, subtract?, min?, max?, floor?, ceil?, round? }`;
- `string` — зарезервировано под DSL-формулу в будущем (в MVP не поддерживается).

Поддерживаемые `ref` в MVP:

- `event.damage`, `event.amount`, `event.stacks`, `event.duration`;
- `intent.damage` (только для модификаторов);
- `source.*`, `target.*`, `eventSource.*`, `eventTarget.*` — характеристики сущностей (`hp`, `maxHp`, `missingHp`, `armor`, `str`, `dex`, `int`, `vit`).

`missingHp` вычисляется как `maxHp - hp`.

Порядок операций фиксирован: `multiply/divide` → `add/subtract` → `min/max` → округление. Если округление не указано, дробные результаты округляются вниз (`floor`).

Для эффекта `MODIFY_DAMAGE`:
- `multiply` может быть как константой, так и `ParametrizedValue`. Внутри слоя все множители (включая параметризованные) применяются перед линейными добавками.
- Добавлено поле `addTags`, позволяющее навешивать теги на интент (например, `attack.crit` для критических ударов).

Открытые вопросы:

- ✅ **Решено:** одно правило порождает ровно один интент. Несколько эффектов — через несколько `ruleIds` у источника. Составные правила отложены на будущее.
- ✅ **Решено для MVP:** параметризация значений эффектов через `ParametrizedValue`. DSL-строки отложены на будущее.

### 3.3. Условия и таргетинг

> ✅ **Решено.**

Условия разделены на два уровня:

- `conditions` — глобальные, проверяются один раз перед выбором целей.
- `targetConditions` — проверяются отдельно для каждой цели, найденной селектором `target`.

Поддерживаемые операторы условий:

- `chance` — шанс срабатывания (по умолчанию per-target в `targetConditions`).
- `hasStatus` — наличие статуса у указанной сущности.
- `statCompare` — сравнение характеристики.
- `distance` — расстояние между сущностями или позициями.
- `faction` — проверка отношения фракций.
- `inTileEffect` — нахождение сущности или позиции в тайловом эффекте.
- `and`, `or`, `not` — логические группы.

Поддерживаемые селекторы целей (`target`):

- `eventTarget`, `eventSource`, `self`, `collisionTarget`;
- `allInRadius` — все сущности в радиусе от заданного центра;
- `nearestEnemy` — ближайший враг в радиусе;
- `chain` — цепочка целей для эффектов вроде цепной молнии.

Разрешение `self`:

| Слой | `self` |
|---|---|
| `source` | Атакующий (обычно `eventSource`). |
| `target` | Цель действия (обычно `eventTarget`). |
| `radius` | Наблюдатель / источник ауры. |
| `world` | Не используется. |

Владелец правила фиксируется в `ownerContext` при добавлении в `activeRules`.

### 3.5a. Контекст правил (`RuleContext`)

> ✅ **Решено.**

`ContentRuleReaction` работает с явным контекстом правил, построенным из события:

```text
RuleContext {
  state: GameState
  event: GameEvent | Intent

  sourceEntityId: EntityId | null
  targetEntityId: EntityId | null

  collisionTargetId: EntityId | null
  abilityTargetPosition: Position | null
  abilityTargets: EntityId[] | null

  eventPosition: Position | null
  eventTags: GameplayTag[]

  eventDamage: number | null
  eventAmount: number | null
  eventDuration: number | null
  eventStacks: number | null
}
```

`eventPosition` строится с fallback: собственная позиция события → позиция цели →
позиция источника → позиция `collisionTargetId` → `null`.

Подробная таблица соответствия событий ↔ полей контекста — в концепте,
раздел 2.2.3.

### 3.6. Циклы реакций

> ✅ **Решено для MVP.**

- Глобальный лимит в **1000 реакций за одну цепочку** принят как достаточный аварийный предохранитель.
- Дополнительные механизмы защиты от циклов (каузальная блокировка, запрет пар правил, журналирование циклов) отложены на пост-MVP.
- Если лимит будет исчерпан, цепочка прерывается. Визуализация и журналирование такой ситуации отложены (см. раздел 3.9).

### 3.7. Статусы как состояния мира

> ✅ Концепт решён. Реализация тайловой части вынесена из MVP в фазу 7. См. раздел «Статусы как состояния мира и тайловые эффекты» в `docs/plans/Концепт боевой системы.md`.

- Статусы делятся на статусы актора (`burning`, `wet`, `oiled`, `dazed`, `stunned`) и тайловые эффекты (`water`, `oil`, `fog`). `oil` может находиться в состоянии `ignited` (горящее масло).
- Производные статусы на акторе (`burning`, `wet`) остаются после выхода из зоны эффекта и тикают по своей длительности.
- В шаблоне статуса задаются поля `mutuallyExclusiveWith`, `blockedBy`, `statusCategory` и `categoryPriority` для системного взаимоисключения, блокировки и разрешения конфликтов в одной фазе.
- При блокировке наложения порождается событие `STATUS_BLOCKED`.
- Тайловые эффекты взаимодействуют с правилами через `ruleIds` в объектах эффектов (реализация — фаза 7).

### 3.8. Окружение

> ✅ Концепт решён. Реализация вынесена из MVP в фазу 7. См. раздел «Тайловые эффекты и окружение» в `docs/plans/Концепт боевой системы.md`.

- Динамические эффекты хранятся в слое `tileEffects`, статические свойства — в шаблонах тайлов.
- Эффекты добавляются/удаляются/трансформируются/поджигаются интентами `SPAWN_TILE_EFFECT`, `REMOVE_TILE_EFFECT`, `TRANSFORM_TILE_EFFECT`, `IGNITE_TILE_EFFECT`.
- Базовые физические переходы выполняются напрямую; сложные реакции — через `ContentRuleReaction`.

### 3.9. Визуализация

> ✅ **Решено.** Детальная проработка вынесена в
> [`docs/plans/Презентационный_слой_и_UI.md`](../plans/Презентационный_слой_и_UI.md).
>
> Кратко: Presentation Layer использует собственное лёгкое состояние отображения
> (`DisplayState`), которое строится последовательным применением патчей от
> событий дерева `ExecutionNode`. UI показывает промежуточные состояния по ходу
> анимаций, а не только финальный результат цепочки реакций.
>
> Конкретные детали реализации (анимации tile-эффектов, точный формат патчей,
> поведение HUD во время анимаций) оставлены на этап реализации и не блокируют
> концепт.

### 3.10. Статусы, наложенные в середине цепочки

> ✅ **Решено для MVP.**

- Правила статуса, наложенного реакцией в процессе цепочки, **не участвуют в текущей цепочке**.
- Правила активируются **сразу после завершения текущей цепочки** (scope — interpretation А1).
- Это означает, что статус активен для всех последующих действий, включая другие фазы в рамках того же `dispatch(action)`.
- Переход к Варианту Б (правила нового статуса участвуют в текущей цепочке на последующих событиях) и вопрос флага `immediate` отложены на пост-MVP.

### 3.11. Интеграция с текущим кодом

> ⚠️ **Низкий приоритет / отложено до финала проработки концепта.**
>
> Эти вопросы касаются реализации в существующей кодовой базе и не блокируют проработку самой механики. Будут решены после финализации типов эффектов, условий и порядка выполнения.

- ✅ **Решено в концепте:** позиция события и контекст правил — через `RuleContext` с fallback на позицию `source` / `target` / `collisionTarget`.
- ✅ **Решено в концепте:** жизненный цикл правил и `activeRules` — через кэш с `ownerContext`.
- ⚠️ **Открыто:** как именно врезать `applyContentModifiers` в `executeIntent`?
- ⚠️ **Открыто:** нужно ли менять тип `Intent`, чтобы у всех интентов был `sourceEntityId`?

### 3.12. Эмерджентные комбо и стихийная система

> ⚠️ **Частично решено.**

**Решено:**

- Выбран подход «состояния + катализаторы»: статусы (`wet`, `oiled`, `burning`) + теги урона (`element.fire`, `element.electric`) дают комбо.
- Зафиксирована стихийная модель взаимодействия статусов `wet` / `oiled` / `burning` и тайловых эффектов `water` / `oil` / `fog`. Огненный урон поджигает `oil`, создавая горящее масло и взрыв; вода тушит горящее масло, образуя пар.
- Взаимоисключение статусов вынесено в системное поведение `APPLY_STATUS` через поля `mutuallyExclusiveWith` и `blockedBy`.
- Добавлены эффект `REMOVE_STATUS` и событие `STATUS_BLOCKED`.
- Зафиксирован минимальный набор физических комбо как контентных правил от оружия:
  - `piercing_crit` — крит для колющего оружия (модификатор на `DAMAGE_INTENT`, множитель зависит от `dex`, добавляет тег `attack.crit`);
  - `blunt_melee_stun` — шанс оглушения при ударе дробящим оружием ближнего боя (длительность зависит от `str`, min 1, max 3);
  - `blunt_push_wall_stun` — оглушение при толчке дробящим оружием в стену, если цель уже `dazed`;
  - `slashing_causes_bleeding` — шанс кровотечения при рубящем уроне.
- Мировые правила описывают базовую физику: `collision_damage`, `collision_daze`.

**Требует обсуждения:**

- Более сложные физические комбо (толчок → столкновение → AoE-ударная волна).
- Кровотечение + движение → дополнительный урон.
- Конкретные нестихийные и нефизические комбо (сами правила и их баланс) — прорабатываются при наполнении контента.
- Инструменты каталогизации и валидации правил.

### 3.13. Предсказуемость vs сюрприз

> ✅ **Решено.**

- Полный превью цепочки реакций **не показывается**.
- Игроку доступна **максимальная ясность правил**: при инспекции экипировки, статусов, талантов, целей (враги/союзники), тайлов и мировых условий видны триггер, условия, шанс, эффект и формула.
- Мировые правила и правила окружения **видны сразу**, без журнала открытий. Идея журнала открытий и скрытых пассивок отложена на post-MVP.
- UI действия показывает только **базовые интенты скилла** (урон, push, наложение статуса и т.д.). Правила не участвуют в превью.
- Combat log фиксирует только **ключевые события** цепочки.

Смысл механики: игрок знает все правила на поле боя, но не знает заранее, как они сложатся в цепочку. Это даёт пространство для экспериментов, просчёта билдов и неожиданных комбинаций без рандома «из ниоткуда».
