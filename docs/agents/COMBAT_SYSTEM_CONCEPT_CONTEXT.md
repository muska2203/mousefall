## 2. Ключевые решения, к которым пришли

### 2.1. Скиллы остаются как сейчас

- У каждого оружия должен быть минимум три скилла:
  - **атакующий** (обычно `1`);
  - **вспомогательный** (обычно `2`);
  - **защитный** (обычно `3`).
- Раскладка `1–3` — ориентировочная, не жёсткая.
- Скиллы должны быть **тупыми генераторами интентов**:
  - проверяют дальность/цель/кулдаун/ОД;
  - создают простые интенты (`DAMAGE`, `PUSH`, `APPLY_STATUS` и т.д.);
  - проставляют **все нужные теги**;
  - не реализуют комбо и не лезут в пассивки.

### 2.2. Два слоя мировых реакций

1. **Системные мировые реакции** — код, инварианты мира:
   - смерть при `hp <= 0`;
   - урон/стан от столкновений;
   - тик статусов;
   - дроп лута;
   - уведомление AI.

2. **Контентные правила через `ContentRuleReaction`** — data-driven правила от предметов, талантов, статусов, окружения.

> **Примечание:** динамические мировые правила (например, временные ауры уровня или глобальные модификаторы сложности) могут появиться позже, но будут прорабатываться отдельно и, скорее всего, реализованы через контентные правила или отдельный механизм, а не через системный реестр реакций.

### 2.3. Две фазы выполнения правил

| Фаза | Когда | Для чего |
|---|---|---|
| **Модификаторы на интенте** | Перед `IntentExecutor` | Изменить урон, дальность, длительность статуса и т.д. |
| **Реакции на событии** | После `IntentExecutor` | Наложить статус, создать дополнительный урон, дать ОД и т.д. |

Порядок модификаторов: сначала по слою `source → target → world → radius`, внутри слоя сначала `multiply`, затем `add`, затем по `priority`, при равных `priority` — по `ruleId`.

Порядок реакций: сначала `ContentRuleReaction` (контентные реакции), затем системные мировые реакции. Интенты, порождённые контентными реакциями, выполняются рекурсивно до системных реакций на исходное событие.

### 2.4. Формат и хранение правила

Каждое правило — отдельный data-driven ассет. Шаблоны предметов, статусов и талантов ссылаются на правила по `ruleIds`.

```text
rule:
  id: slashing_causes_bleeding
  ownerContext:
    type: entity
    entityId: abc-123
  trigger:
    event: ENTITY_DAMAGED
    tags: [damage.physical.slashing]
  targetConditions:
    - type: chance
      value: 0.3
  effect:
    type: APPLY_STATUS
    status: bleeding
    duration: 3
  target: eventTarget
  priority: 0
```

Основные поля:

- `id` — уникальный идентификатор ассета правила.
- `ownerContext` — владелец правила в рантайме (заполняется при добавлении в `activeRules`).
- `trigger.event` — событие или интент (`ENTITY_DAMAGED`, `DAMAGE_INTENT`, `ENTITY_HEALED` и т.д.).
- `trigger.tags` — обязательные теги.
- `conditions` — глобальные JSON-операторы, проверяемые один раз: `chance`, `hasStatus`, `statCompare`, `distance`, `faction`, `inTileEffect`, `and`, `or`, `not`.
- `targetConditions` — JSON-операторы, проверяемые отдельно для каждой цели, найденной селектором `target`.
- `effect` — что делает правило.
- `target` — селектор целей (`eventTarget`, `eventSource`, `self`, `allInRadius`, `nearestEnemy`, `chain`).
- `priority` — порядок срабатывания.

### 2.5. Откуда собирать правила

Правила собираются из четырёх источников:

1. **Источник действия** — кто совершил действие.
2. **Цель действия** — на ком оно отразилось.
3. **Мир / окружение** — тайлы, лужи, туман, горящие клетки.
4. **Сущности в радиусе события** — для аур и наблюдателей.

Разрешение `self` зависит от слоя источника:

| Слой | `self` разрешается в |
|---|---|
| `source` | Атакующий (обычно `eventSource`). |
| `target` | Цель действия (обычно `eventTarget`). |
| `radius` | Сторонний наблюдатель / источник ауры. |
| `world` | Не используется; правила мира работают с `eventSource`, `eventTarget`, `eventPosition`. |

Пример наблюдателя:

```text
trigger:
  event: ENTITY_HEALED
conditions:
  - type: distance
    from: self
    to: eventTarget
    max: 3
  - type: faction
    target: eventTarget
    relation: enemy
    to: self
effect:
  type: APPLY_STATUS
  status: rage
  duration: 2
target: self
```

> **Примечание:** здесь `self` — это владелец правила (наблюдатель), а не источник или цель события. Правило срабатывает, когда враг лечится в радиусе 3 клеток от наблюдателя, и накладывает `rage` на самого наблюдателя.

### 2.6. Жизненный цикл правил

- Шаблоны предметов, статусов и талантов содержат `ruleIds`; сами правила хранятся в отдельных ассетах.
- Рекомендуется **кэшировать активные правила у актора** (`actor.activeRules`).
- При надевании/снятии предмета, наложении/снятии статуса, получении/отзыве таланта правила добавляются/удаляются.
- Каждое правило помечается источником через `ownerContext` (`itemInstanceId`, `statusInstanceId`, `abilityId`), чтобы его можно было корректно удалить и разрешить `self`.
- Правила окружения не кэшируются у актора, а определяются по позиции.

### 2.7. Один интент на правило

> ✅ **Решено:** одно правило описывает ровно один шаблон намерения.

- Каждое правило имеет ровно один `effect`, который описывает один шаблон `Intent`.
- Если источнику (предмет, статус, талант) нужно несколько разных эффектов, в его шаблоне перечисляется несколько `ruleIds`.
- Порядок выполнения таких эффектов определяется `priority` и `ruleId` внутри фазы.
- Условия правила (`chance`, `hasStatus` и т.д.) проверяются отдельно для каждого правила.
- При использовании мультитаргетного селектора (`allInRadius`, `chain`) один шаблон выполняется по одному разу для каждой подходящей цели, порождая несколько конкретных инстансов `Intent`.
- Составные правила, порождающие несколько разных типов интентов за одно срабатывание, сознательно отложены. Если они понадобятся, для них будет введён отдельный механизм.

### 2.8. Ограничения

- Вероятности должны использовать детерминированный рандом.
- Ограничение на глубину цепочки реакций — например, **1000 реакций за один шаг**.
- Модификаторы должны быть чистыми: меняют только значения в интенте, не мутируют состояние мира.
- Контентные реакции выполняются перед системными мировыми реакциями.
- Порядок правил: `source → target → world → radius`, затем `priority`, затем `ruleId` (для модификаторов внутри слоя `multiply` перед `add`).
- Правило срабатывает один раз на объект события, но может срабатывать многократно в рамках одной цепочки, если происходит несколько подходящих событий.

### 2.9. Тайловые эффекты и окружение

- Динамические эффекты окружения хранятся в отдельном слое `tileEffects` в `GameState`.
- В одной клетке может быть несколько эффектов разных типов, но только один эффект каждого типа; новый экземпляр обновляет длительность/интенсивность.
- Тайловые эффекты тикают в `round-recovery` / на `BEGIN_TURN` нового раунда.
- Правила окружения хранятся в ассетах тайловых эффектов и шаблонах тайлов; собираются `ContentRuleReaction` по позиции события.
- Базовые физические переходы (огонь + вода → пар, огонь + масло → горящее масло) выполняются напрямую и порождают событие `TILE_EFFECT_CHANGED`.
- Огонь поджигает масло в радиусе 1; масло, появившееся рядом с огнём, тоже загорается.
- Стартовый набор эффектов: `fire`, `water`, `oil`, `fog`.

## 3. Открытые вопросы и проблемы

Ниже список вопросов, которые ещё не проработаны. Их нужно обсуждать и решать.

### 3.1. Формат хранения правил

> ✅ Решено.

- Правила хранятся в отдельных ассетах.
- Шаблоны предметов, статусов и талантов ссылаются на правила по `ruleIds`.
- Для условий используется декларативный JSON; отдельный DSL откладывается.

### 3.2. Типы эффектов

> ✅ Core-набор решён.

Минимальный набор для старта:

| Эффект | Фаза | Описание | Параметризуемые поля |
|---|---|---|---|
| `MODIFY_DAMAGE` | Модификаторы на интенте | Изменяет урон интента | `add` (`multiply` — только константа) |
| `APPLY_STATUS` | Реакции на событии | Накладывает статус на цель | `duration`, `stacks` |
| `DEAL_DAMAGE` | Реакции на событии | Наносит дополнительный урон | `amount` |
| `HEAL` | Реакции на событии | Восстанавливает здоровье | `amount` |
| `RESTORE_AP` | Реакции на событии | Восстанавливает очки действия | — |
| `CONSUME_AP` | Реакции на событии | Тратит очки действия у владельца правила | `amount` |

Числовые поля эффектов задаются через `ValueSpec`:

- `number` — константа;
- `ParametrizedValue` — объект `{ ref, multiply?, divide?, add?, subtract?, min?, max?, floor?, ceil?, round? }`;
- `string` — зарезервировано под DSL-формулу в будущем (в MVP не поддерживается).

Поддерживаемые `ref` в MVP:

- `event.damage`, `event.amount`, `event.stacks`, `event.duration`;
- `intent.damage` (только для модификаторов);
- `source.*`, `target.*`, `eventSource.*`, `eventTarget.*` — характеристики сущностей (`hp`, `maxHp`, `missingHp`, `armor`, `str`, `dex`, `int`, `vit`).

`missingHp` вычисляется как `maxHp - hp`.

Порядок операций фиксирован: `multiply/divide` → `add/subtract` → `min/max` → округление. Если округление не указано, дробные результаты округляются вниз (`floor`).

Открытые вопросы:

- ✅ **Решено:** одно правило порождает ровно один интент. Несколько эффектов — через несколько `ruleIds` у источника. Составные правила отложены на будущее.
- ✅ **Решено для MVP:** параметризация значений эффектов через `ParametrizedValue`. DSL-строки отложены на будущее.

### 3.3. Условия и таргетинг

> ✅ **Решено.**

Условия разделены на два уровня:

- `conditions` — глобальные, проверяются один раз перед выбором целей.
- `targetConditions` — проверяются отдельно для каждой цели, найденной селектором `target`.

Поддерживаемые операторы условий:

- `chance` — шанс срабатывания (по умолчанию per-target в `targetConditions`).
- `hasStatus` — наличие статуса у указанной сущности.
- `statCompare` — сравнение характеристики.
- `distance` — расстояние между сущностями или позициями.
- `faction` — проверка отношения фракций.
- `inTileEffect` — нахождение сущности или позиции в тайловом эффекте.
- `and`, `or`, `not` — логические группы.

Поддерживаемые селекторы целей (`target`):

- `eventTarget`, `eventSource`, `self`;
- `allInRadius` — все сущности в радиусе от заданного центра;
- `nearestEnemy` — ближайший враг в радиусе;
- `chain` — цепочка целей для эффектов вроде цепной молнии.

Разрешение `self`:

| Слой | `self` |
|---|---|
| `source` | Атакующий (обычно `eventSource`). |
| `target` | Цель действия (обычно `eventTarget`). |
| `radius` | Наблюдатель / источник ауры. |
| `world` | Не используется. |

Владелец правила фиксируется в `ownerContext` при добавлении в `activeRules`.

### 3.4. Производительность

> ✅ **Частично решено для MVP.**

- **Поиск наблюдателей в радиусе:** в MVP используется прямое сканирование всех сущностей с проверкой расстояния. Это допустимо на небольших картах и при десятках сущностей.
- **Пространственный индекс:** не нужен в MVP; может быть добавлен позже при росте масштаба.
- **Размер `activeRules`:** ограничен количеством экипировки, статусов и талантов актора. Конкретный лимит будет определяться балансом и тестированием.

- Нужен ли жёсткий лимит на количество правил у одного актора?

### 3.5. Порядок выполнения

> ✅ Решено.

- Контентные реакции (`ContentRuleReaction`) выполняются перед системными мировыми реакциями.
- Порожденные контентными реакциями интенты выполняются рекурсивно до запуска системных реакций на исходное событие.
- Внутри одного блока правила сортируются по `priority` (меньше — раньше), при равных `priority` — по `ruleId`.
- Правила собираются в порядке источников: `source → target → world → radius`.
- Правило срабатывает один раз на объект события, но может срабатывать многократно в рамках одной цепочки, если происходит несколько подходящих событий.

### 3.7. Статусы как состояния мира

> ✅ Решено. См. раздел «Статусы как состояния мира и тайловые эффекты» в `docs/plans/Концепт боевой системы.md`.

- Статусы делятся на статусы актора (`burning`, `wet`, `oiled`) и тайловые эффекты (`fire`, `water`, `oil`, `fog`).
- Производные статусы на акторе (`burning`, `wet`) остаются после выхода из зоны эффекта и тикают по своей длительности.
- Тайловые эффекты взаимодействуют с правилами через `ruleIds` в ассетах эффектов.

### 3.8. Окружение

> ✅ Решено. См. раздел «Тайловые эффекты и окружение» в `docs/plans/Концепт боевой системы.md`.

- Динамические эффекты хранятся в слое `tileEffects`, статические свойства — в шаблонах тайлов.
- Эффекты добавляются/удаляются/трансформируются интентами `SPAWN_TILE_EFFECT`, `REMOVE_TILE_EFFECT`, `TRANSFORM_TILE_EFFECT`.
- Базовые физические переходы выполняются напрямую; сложные реакции — через `ContentRuleReaction`.
