# Жизненный цикл правил и `activeRules`

> Подробное описание того, как правила от предметов, статусов и талантов
> становятся активными, кэшируются у актора и удаляются.
> `activeRules` — производный кэш, первичными источниками остаются
> экипировка, статусы и таланты.

---

## 1. Общий принцип

Правила живут на источниках:

- **Экипировка** — предметы в слотах `weapon` / `armor` / `amulet`.
- **Статусы** — эффекты, висящие на акторе (`burning`, `wet`, `stunned` и т.д.).
- **Таланты** — пассивные и активные способности, полученные актором.
- **Окружение** — глобальные мировые правила, тайловые эффекты и свойства тайлов.

Правила от экипировки, статусов и талантов **кэшируются** у актора в массиве
`activeRules`. Это ускоряет боевой цикл и упрощает поиск наблюдателей в радиусе
события.

Правила окружения **не кэшируются** у актора. Они собираются в момент события
по позиции `eventPosition`.

---

## 2. Структура `ActiveRule` и `OwnerContext`

Каждая запись в `activeRules` — это не просто `ContentRule`, а `ActiveRule`:

```text
ActiveRule {
  rule: ContentRule
  ownerContext: OwnerContext
}
```

`OwnerContext` определяет, от чьего имени сработает правило, и нужен для двух
вещей:

1. **Корректного удаления** правил при пропадании источника.
2. **Разрешения `self`** внутри условий и эффектов правила.

```text
OwnerContext {
  type: 'entity' | 'tileEffect'

  # Для правил от экипировки, статуса или таланта
  entityId?: EntityId

  # Для статусов. Если на акторе может быть несколько экземпляров
  # одного статуса, у каждого свой instanceId.
  statusInstanceId?: string

  # Для тайловых эффектов
  position?: Position
  tileEffectType?: string
}
```

---

## 3. Экипировка

### При надевании предмета (`EQUIP_ITEM`)

1. Найти шаблон предмета по `templateId`.
2. Для каждого `ruleId` в шаблоне:
   - Найти `ContentRule` по `ruleId`.
   - Создать `ActiveRule` с `ownerContext.type = 'entity'` и
     `ownerContext.entityId = itemInstanceId`.
   - Добавить запись в `actor.activeRules`.

### При снятии предмета (`UNEQUIP_ITEM`)

Удалить из `actor.activeRules` все записи, где
`ownerContext.entityId === itemInstanceId`.

### Важные нюансы

- `itemInstanceId` — ID конкретного экземпляра предмета, а не `templateId`.
  Два одинаковых кольца дают два независимых набора правил.
- Если предмет меняет слот, эмулируется `UNEQUIP_ITEM` + `EQUIP_ITEM`.

---

## 4. Статусы

### При наложении статуса (`STATUS_APPLIED`)

1. Найти шаблон статуса.
2. Сгенерировать уникальный `statusInstanceId` для этого экземпляра статуса.
3. Для каждого `ruleId` в шаблоне:
   - Создать `ActiveRule` с `ownerContext.type = 'entity'` и
     `ownerContext.entityId = statusInstanceId`.
   - Добавить запись в `actor.activeRules`.

### При снятии статуса (`STATUS_REMOVED`)

Удалить из `actor.activeRules` все записи, где
`ownerContext.entityId === statusInstanceId`.

### Зачем нужен `statusInstanceId`

Если на акторе может висеть несколько экземпляров одного статуса, у каждого
должен быть свой `statusInstanceId`. Иначе при снятии одного стака мы удалим
правила второго.

Даже если в MVP статусы одного типа не стакаются, а только обновляют
длительность, `statusInstanceId` нужен, чтобы правила удалялись при полном
исчезновении статуса, а не при обновлении длительности.

### Идентификатор статуса

В MVP используется UUID, сгенерированный при наложении статуса.
`activeRules` не сохраняется, поэтому стабильность идентификатора между
сессиями не требуется.

---

## 5. Таланты

### При получении таланта (`GRANT_ABILITY`)

1. Найти шаблон таланта.
2. Для каждого `ruleId` в шаблоне:
   - Создать `ActiveRule` с `ownerContext.type = 'entity'` и
     `ownerContext.entityId = abilityId + ':' + sourceItemInstanceId`.
   - Добавить запись в `actor.activeRules`.

### При отзыве таланта (`REVOKE_ABILITY`)

Удалить все записи, где `ownerContext.entityId` начинается с `abilityId + ':'`.

### Почему `abilityId + sourceItemInstanceId`

Один и тот же `abilityId` может быть получен из нескольких источников —
например, от двух разных предметов или от предмета и уровня. Разные источники
— разные записи в `activeRules`, даже для одного `abilityId`.

---

## 6. Окружение

Правила окружения **не кэшируются** у актора. Они собираются в момент события
из трёх источников:

1. **Глобальные мировые правила** — жёстко заданный набор, активен всегда.
2. **Тайловые эффекты** — из `tileEffects` в позиции события. Для каждого
   эффекта создаётся `ActiveRule` с `ownerContext.type = 'tileEffect'`,
   `position` и `tileEffectType`.
3. **Статические свойства тайла** — из шаблона тайла в позиции события.

Эти правила живут только на время обработки события.

---

## 7. Валидация шаблонов

На этапе загрузки контента (или в CI / тестах) проверяется:

1. Все `ruleIds`, на которые ссылаются шаблоны, существуют в реестре правил.
2. В одном шаблоне нет дублирующихся `ruleIds`.
3. Правила не ссылаются на несуществующие статусы, таланты или эффекты.

Если в будущем понадобится, чтобы один предмет давал один и тот же эффект
несколько раз, это решается не дублированием `ruleId`, а отдельным механизмом
составных правил.

---

## 8. Обновление `activeRules` в процессе цепочки реакций

Изменения состояния применяются **немедленно**:

- Если статус наложен — его правила добавляются в `activeRules`.
- Если статус снят — его правила удаляются из `activeRules`.

Но текущая фаза `ContentRuleReaction` работает со **снимком** правил,
собранным в начале фазы. Новые правила в этот снимок не попадают.

После завершения цепочки снимок сбрасывается, и следующая фаза работает с
актуальным `activeRules`.

Это сохраняет решение о том, что статусы, наложенные в середине цепочки,
не участвуют в текущей цепочке, но не откладывает обновление состояния.

---

## 9. Сохранения и загрузка

`activeRules` **не сохраняется** в save-файл. При загрузке игры он
пересобирается:

1. Для каждого актора пройтись по экипировке → добавить правила.
2. Пройтись по статусам → добавить правила.
3. Пройтись по талантам → добавить правила.

Это исключает рассинхронизацию и уменьшает размер сохранений.

---

## 10. Производительность

В MVP достаточно линейного сканирования `activeRules` при каждом событии.
Для ускорения можно ввести простой индекс по `trigger.event`: разделить
`activeRules` на подмножества по типу события / интента, чтобы не сканировать
все правила.

Пространственный индекс для наблюдателей в радиусе откладывается на пост-MVP.
На небольших картах прямое сканирование всех сущностей допустимо.

---

## 11. Разрешение `self` через `ownerContext`

| Слой источника | `self` разрешается в |
|---|---|
| `source` | Владелец правила в `activeRules` актора-источника |
| `target` | Владелец правила в `activeRules` актора-цели |
| `radius` | Владелец правила в `activeRules` наблюдателя |
| `world` | Не используется |

**Пример.** Правило `fire_aura_ignites_enemies` висит на статусе игрока. Когда
начинается ход врага, движок находит это правило в слое `radius` (игрок стоит
рядом). `self` внутри правила разрешается в игрока. Эффект
`APPLY_STATUS burning` применяется к врагам в радиусе от игрока.

---

## 12. Чеклист

- [ ] `activeRules` — производный кэш, не сохраняется в save.
- [ ] Каждый `ActiveRule` имеет `ownerContext`.
- [ ] Экипировка: добавление / удаление по `itemInstanceId`.
- [ ] Статусы: добавление / удаление по `statusInstanceId`.
- [ ] Таланты: добавление / удаление по `abilityId + sourceItemInstanceId`.
- [ ] Окружение: не кэшируется, собирается по позиции события.
- [ ] Валидация шаблонов: существование `ruleIds`, отсутствие дублей в одном
      шаблоне.
- [ ] Изменения `activeRules` применяются немедленно, но текущая фаза работает
      со снимком.
- [ ] `self` разрешается через `ownerContext`, а не через `eventSource` /
      `eventTarget`.
