# Операционный план выполнения фазы 6 «Полировка и пост-MVP»

> Статус: черновик плана, подготовлен системным аналитиком / синьор-разработчиком.
> Текущее состояние проекта: фаза 5.5 «Адаптация Presentation» в процессе, фаза 6 не начата.
> База: `concept/combat-system`, commit `ba026f7`.

---

## 1. Контекст и ограничения

### 1.1. Что уже реализовано (база для полировки)

- Инфраструктура декларативных content-rules:
  - `RuleContext`, `ContentRuleReaction`, `activeRules`, жизненный цикл правил.
  - Реестр `CONTENT_RULES` / `WORLD_CONTENT_RULES`, `registry.ts`, базовая `validation.ts`.
  - Модификаторы на `DAMAGE`-интентах, защита от циклов (лимит 1000).
- Пилотные мировые правила: горение, тик горения, столкновения, контратака.
- Presentation-слой: `ExecutionNode`, combat log, базовые анимации, debug-режим (читы + карта).
- Тестовое покрытие content-rules высокое (~95 % statements).

### 1.2. Главные пробелы

1. **Валидация контента не покрывает CI.** `npm run build` не загружает `public/content`, `deploy.yml` не запускает тесты / `typecheck` / `validate:i18n`.
2. **Content-rules невидимы для Presentation.** Нет события `RULE_TRIGGERED` в `ExecutionNode`, нет debug-визуализации цепочек.
3. **Реальный контент почти не использует правила.** Только `counterattack.json` имеет `ruleIds`; предметы и способности — пустые.
4. **Combat log минималистичный.** Нет источника событий, нет типов урона, нет группировки.
5. **Нет системной производственной нагрузочной проверки** типичных боевых цепочек.
6. **Таланты как отдельный слой контента отсутствуют.** В концепте упоминаются «таланты», в коде их нет (есть только `AbilityTemplate`).

### 1.3. Принципы выполнения фазы 6

- **Не ломать старое:** feature flag `contentRulesEnabled` сохраняется до явного переключения.
- **Тесты — часть задачи:** каждый рабочий пакет включает unit-/integration-тесты и обновление покрытия.
- **Документация обновляется одновременно с кодом:** `AGENTS.md`, `docs/agents/`, README при необходимости.
- **Минимальные изменения:** не рефакторить ради рефакторинга; цель — релиз-кандидат MVP.
- **Data-driven контент:** системные реакции остаются кодом, контентные реакции — декларативными.

## 2. Рабочие пакеты (WP)

### WP6.1 — Валидация шаблонов и правил + CI

**Цель:** любая битая ссылка в контенте ломает сборку/тесты.

**Задачи:**

1. **Расширить `validation.ts`**:
   - Проверка, что `applyStatus.statusType` существует в `statuses`.
   - Проверка, что `counterAttack.skillId` / `dealDamage.damageFormulaId` / `heal.healFormulaId` существуют (если применимо).
   - Проверка, что теги в `trigger.tags` / `condition.tag` не пустые и консистентны (предупреждение, если тег не известен).
2. **Создать Node-скрипт `scripts/validate-content.ts`**:
   - Загружает `public/content/manifest.json` через `fs`.
   - Парсит все JSON-шаблоны (`items`, `statuses`, `abilities`, `entities`, `players`, `maps`, `doors`, `stairs`).
   - Вызывает `validateContentRuleReferences` и новые перекрёстные проверки.
   - Проверяет, что у каждого content ID есть перевод в `src/content/texts/{ru,en}/`.
   - Выходит с кодом `1` при ошибках, печатает понятный отчёт.
3. **Добавить `package.json` скрипты**:
   - `"validate:content": "tsx scripts/validate-content.ts"`
   - `"validate": "npm run typecheck && npm run validate:i18n && npm run validate:content"`
4. **Обновить `.github/workflows/deploy.yml`**:
   - Добавить job `quality` перед `deploy`:
     1. `npm ci`
     2. `npm run validate`
     3. `npm test`
   - Job `deploy` зависит от `quality` (`needs: quality`).
5. **Исправить существующие проблемы контента**, если валидация их найдёт:
   - Отсутствующие тексты (`cat_guardian_plate`, `cat_guardian_maul`, `unarmed`, `cat_guardian`).
   - Пустые `ruleIds` — не ломать, но зафиксировать в бэклоге WP6.3.

**Критерий готовности:**

- `npm run validate:content` проходит на `public/content`.
- CI job `quality` проходит на `main`.
- Добавление несуществующего `ruleId` или битой ссылки ломает `npm run validate`.

**Тесты:**

- Unit-тесты на новые валидаторы.
- Интеграционный тест `scripts/validate-content.ts` через `execa` / `child_process`.

**Зависимости:** нет (можно начинать параллельно с WP6.2 foundation и WP6.5).

### WP6.2 — Отладочная визуализация цепочек правил

**Цель:** в debug-режиме видны сработавшие content-rules и порождённые ими интенты.

**Задачи:**

1. **Simulation: событие `RULE_TRIGGERED`.**
   - Добавить тип события `RULE_TRIGGERED` в `GameEvent` (`src/simulation/core-types.ts`).
   - Поля: `ruleId`, `layer` (`source` | `target` | `world` | `radius`), `ownerEntityId`, `triggerEventType`, `triggerTags`, `intents: Intent[]`, `conditionMatched`.
   - В `content-rule-reaction.ts` удалить `void builder; void parent;` и записывать `RULE_TRIGGERED` в дерево `ExecutionNode` как child текущего узла.
   - Убедиться, что `RULE_TRIGGERED` не влияет на игровую логику (только observability).
2. **Presentation: combat log.**
   - В `logBuilder.ts` добавить обработку `RULE_TRIGGERED` — печатать локализованное сообщение вида «[DEBUG] Сработало правило {ruleId} ({layer}) → {intents}».
   - Добавить цветовое кодирование по слою.
3. **Presentation: debug-оверлей.**
   - В `DebugPanel.tsx` добавить секцию «Последние реакции» — список 10 последних `RULE_TRIGGERED` из текущего хода.
   - Добавить toggle «Показывать цепочки правил в логе».
4. **Анимация:**
   - При `RULE_TRIGGERED` в debug-режиме рисовать краткую вспышку/текст над владельцем правила (`UI_FLOATING_TEXT` с `ruleId`).

**Критерий готовности:**

- При включённом debug-режиме (`` ` ``) в combat log и в DebugPanel видны сработавшие content-rules.
- В `ExecutionNode` дереве `RULE_TRIGGERED` является потомком события, его вызвавшего.
- Все тесты проходят; новые unit-тесты покрывают `RULE_TRIGGERED`.

**Тесты:**

- Unit: `content-rule-reaction.test.ts` — проверить, что `RULE_TRIGGERED` эмитится.
- Unit: `logBuilder.test.ts` — проверить debug-сообщение.
- Integration: end-to-end бой с контратакой/горением → в логе/дереве видны правила.

**Зависимости:**

- Можно начинать foundation (событие + запись в дерево) параллельно с WP6.1.
- Presentation-часть зависит от завершения фазы 5.5 (адаптация Presentation), но `RULE_TRIGGERED` можно добавить без блокировки.

### WP6.3 — Балансировка и каталогизация стартовых правил

**Цель:** задокументированный и сбалансированный набор стартовых правил; дизайн утверждает цифры.

**Задачи:**

1. **Аудит существующего контента:**
   - Составить таблицу всех `ruleIds` в `public/content/` и в коде (`global-rules.ts`, `counterattack-rules.ts`).
   - Для каждого правила: источник, триггер, условие, эффект, текущие числа.
2. **Каталог стартовых правил:**
   - Создать документ `docs/design/starting-rules-catalog.md`.
   - Описать 8–12 стартовых правил: базовое оружие, кольца/амулеты, статусы (горение, яд, контратака, оглушение), базовые комбо.
   - Зафиксировать числа: шансы, длительности, множители, базовый урон, AP.
3. **Наполнение контента:**
   - Добавить `ruleIds` в JSON-предметы (`dagger_of_venom`, `flaming_sword`, `ring_of_counter` и т.д. — по каталогу).
   - Добавить/обновить JSON-статусы, если нужны новые эффекты.
   - Убедиться, что тексты и переводы добавлены.
4. **Боевые сценарии:**
   - Написать 3–5 интеграционных тестов-сценариев (`tests/integration/combat-scenarios/`):
     - игрок с огненным мечом против крыс;
     - игрок с ядовитым кинжалом + контратака;
     - столкновение + горение + тик урона.
   - Сценарии должны проходить и не ломать баланс (враги не умирают за один ход, игрок не умирает мгновенно).
5. **Утверждение дизайном:**
   - В документе отметить статус «Утверждено» после проверки.

**Критерий готовности:**

- `docs/design/starting-rules-catalog.md` содержит согласованный набор правил и чисел.
- Реальный контент использует правила из каталога.
- Боевые сценарии проходят.

**Тесты:**

- Новые integration-тесты combat-сценариев.
- Обновление `content-rule-sources.test.ts` при добавлении новых источников.

**Зависимости:**

- WP6.1 должен быть готов, чтобы валидация новых `ruleIds` работала.
- WP6.2 foundation желателен, но не обязателен.

### WP6.4 — Документирование крайних случаев

**Цель:** новый разработчик или агент может по документам добавить правило и предсказать его поведение.

**Задачи:**

1. **Создать `docs/agents/CONTENT_RULES_EDGE_CASES.md`:**
   - Self-эффекты (`self` селектор, `source === target`).
   - Циклы: лимит 1000, как он работает, как отлаживать.
   - Mid-chain статусы: статус, наложенный реакцией, участвует ли в дальнейших реакциях текущей цепочки.
   - Конфликты статусов: `mutuallyExclusiveWith`, `blockedBy`, замена vs стаки.
   - Пустые селекторы: что произойдёт, если `targetSelector` не нашёл целей.
   - Условия модификаторов (когда будут реализованы — см. пост-MVP) и их текущее поведение.
   - Порядок слоёв: `source` → `target` → `world` → `radius`, сортировка внутри слоя.
   - Параметризованные значения: `multiply`, `min`, `round`, `context`.
2. **Обновить `docs/agents/INDEX.md`:**

3. **Обновить `src/simulation/content-rules/AGENTS.md`:**

4. **Обновить `docs/agents/CONTENT.md`:**
   - Упомянуть `statuses` и `texts/` структуру.

**Критерий готовности:**

- Документы актуальны и соответствуют коду.

**Тесты:** не требуются (документация).

**Зависимости:** WP6.1–WP6.3.

### WP6.5 — Проверка производительности

**Цель:** понять, есть ли узкие места в content-rules на целевых картах; если есть — оформить пост-MVP задачи.

**Задачи:**

1. **Целевые сценарии:**
   - Карта 20×20, 10 врагов, каждый с 3–5 active rules.
   - AoE-скилл, задевающий 5 целей.
   - Цепочка горения + столкновение + тик статусов.
2. **Benchmark-harness:**
   - Создать `tests/perf/content-rules.perf.test.ts` (vitest bench или ручной замер).
   - Замеряет время выполнения `GameSimulation.dispatch` для типичной цепочки.
   - Считает количество срабатываний правил за цепочку.
3. **Измерения:**
   - Время на один intent + реакции.
   - Время на сбор `activeRules` (кэш vs rebuild).
   - Время на селектор `allInRadius` / `nearestEnemy`.
   - Влияние `RULE_TRIGGERED` событий на производительность (debug vs release).
4. **Отчёт:**
   - Создать `docs/plans/combat-implementation-roadmap/06-polirovka-perf-report.md`.
   - Если показатели в пределах нормы — зафиксировать baseline.
   - Если есть узкие места — описать их и внести в пост-MVP бэклог (WP6.6).

**Критерий готовности:**

- Benchmark запускается через `npm test` или отдельный скрипт.
- На целевых картах нет заметных фризов (замер < 16 мс на intent на среднем ноутбуке разработчика).
- Отчёт зафиксирован.

**Тесты:**

- Perf-тесты (не должны быть flaky; использовать детерминированный seed).

**Зависимости:** нет (можно начинать параллельно с WP6.1).

### WP6.6 — Формирование пост-MVP бэклога

**Цель:** приоритизированный список задач после MVP, согласованный с архитектурой.

**Задачи:**

1. **Собрать открытые вопросы** из всех предыдущих WP:
   - Защита от циклов beyond лимита 1000 (каузальная блокировка, журналирование).
   - DSL-формулы для параметризации значений.
   - Составные правила (несколько интентов за одно срабатывание).
   - Журнал открытий / скрытые правила.
   - Пространственный индекс для наблюдателей в радиусе.
   - Полное превью цепочки.
   - Вариант Б для mid-chain статусов.
   - Глобальная нормализация `Intent` (общие поля `sourceEntityId`/`targetEntityId`/`tags`).
   - Тайловые эффекты и правила окружения.
   - Условия на модификаторах.
   - Расширение модификаторов за пределы `DAMAGE`.
2. **Приоритизация:**
   - P0 — критично для стабильности (циклы, нормализация Intent).
   - P1 — сильно улучшает геймплей (тайловые эффекты, составные правила, DSL).
   - P2 — качество жизни (журнал открытий, полное превью).
3. **Документ:**
   - Создать `docs/plans/combat-implementation-roadmap/post-mvp-backlog.md`.
   - Каждая задача: описание, мотивация, примерное решение, связь с концептом.

**Критерий готовности:**

- Бэклог задокументирован, приоритизирован и не содержит критических дыр.

**Тесты:** не требуются.

**Зависимости:** WP6.1–WP6.5.
