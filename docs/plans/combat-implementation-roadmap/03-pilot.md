# Фаза 3. Пилот

## Цель фазы

Впервые пропустить реальное боевое событие через новую систему правил, убедиться, что цепочка работает, и сохранить возможность отката к старой системе. Пилот должен быть узким: **одно событие, 1–2 простых правила**.

---

## Шаги

### Шаг 3.1. Выбор пилотного сценария

- **Вход:** каркасы из фазы 2, решения из decision log.
- **Результат:** зафиксированный пилот в два шага:
  1. Реакция на событие: «огненный урон → горение» (на `ENTITY_DAMAGED` с тегом `damage.magical.fire`).
  2. Модификатор на интенте: «огненный урон ×1,5» (на `DAMAGE_INTENT` с тем же тегом).
- **Критерий готовности:** выбор документирован, понятна последовательность шагов, риски учтены; Presentation/UI в пилоте не затрагиваются.
- **Зависимости:** Фаза 2.
- **Риски:**
  - Для модификатора огня нужно временно изолировать старую `fireDamageReaction`, иначе урон дублируется.
  - Реакция на огонь может конфликтовать с существующей обработкой горения, если та тоже накладывает `burning`.
- **Старая / новая система:** новые правила добавляются в реестр; старая система пока не отключается, кроме случая огненного модификатора.

---

### Шаг 3.2. Включение `ContentRuleReaction` для пилота

- **Вход:** выбранная реакция «огненный урон → горение», построитель `RuleContext`, каркас реакций.
- **Результат:**
  - при включённом флаге после выполнения пилотного `DAMAGE`-интента `ContentRuleReaction` проверяет правила;
  - подходящее правило с шансом порождает `APPLY_STATUS burning`;
  - порождённые интенты проходят через тот же боевой цикл;
  - при выключенном флаге старые реакции остаются единственными.
- **Критерий готовности:** unit-тест и ручная проверка показывают цепочку: скилл → урон → событие → реакция → горение.
- **Зависимости:** Шаг 3.1.
- **Риски:**
  - `RuleContext` для `ENTITY_DAMAGED` построен неверно. *Как заметить:* правило не срабатывает или срабатывает на не той цели.
  - Статус `burning` не учитывает `blockedBy`/`mutuallyExclusiveWith`. *Как заметить:* конфликты статусов разрешаются неверно.
- **Старая / новая система:** новая реакция активна только для пилота; Presentation/UI в пилоте не затрагиваются.

---

### Шаг 3.3. Включение модификатора на интенте для пилота

- **Вход:** выбранное правило-модификатор «огненный урон ×1,5», точка включения из фазы 2, решение по изоляции старой `fireDamageReaction`.
- **Результат:**
  - при включённом флаге `DAMAGE_INTENT` с тегом `damage.magical.fire` проходит через новый модификатор;
  - урон изменяется согласно правилу;
  - `IntentExecutor` применяет уже изменённый интент;
  - старая `fireDamageReaction` временно не срабатывает для этого случая, чтобы избежать дублирования;
  - при выключенном флаге интент остаётся неизменным и работает старая реакция.
- **Критерий готовности:** unit-тесты показывают, что с флагом урон огня изменяется, без флага — нет; guardian tests проходят в обоих режимах.
- **Зависимости:** Шаг 3.2.
- **Риски:**
  - Модификатор срабатывает для всех `DAMAGE_INTENT`, а не только для огня. *Как заметить:* тесты на другие типы урона показывают неожиданные изменения.
  - Модификатор меняет порядок операций. *Как заметить:* итоговый урон не совпадает с ожидаемым.
  - Старая `fireDamageReaction` не изолирована. *Как заметить:* огненный урон дублируется.
- **Старая / новая система:** новая ветка обработки активна только для пилотного случая; старая `fireDamageReaction` временно обходится.

---

### Шаг 3.4. Детерминированный рандом и защита от циклов

- **Вход:** правила с шансом, новый боевой цикл.
- **Результат:**
  - вероятности используют детерминированный runtime random;
  - глобальный лимит в 1000 реакций за цепочку прерывает бесконечные циклы;
  - при исчерпании лимита цепочка останавливается без зависания;
  - информация об исчерпании лимита выводится в консоль.
- **Критерий готовности:** повторный запуск одного и того же состояния даёт тот же результат; тест с искусственным циклом обрывается на лимите.
- **Зависимости:** Шаг 3.3.
- **Риски:**
  - Разные seed приводят к непредсказуемому поведению тестов. *Как заметить:* тесты на шанс плавают.
  - Лимит слишком жёсткий или слишком мягкий. *Как заметить:* либо обрываются легальные длинные цепочки, либо зависает тест.
- **Старая / новая система:** новая система отвечает за рандом и лимит в своей ветке; старая система сохраняет своё поведение.

---

### Шаг 3.5. Тесты пилотной цепочки

- **Вход:** фикстуры, пилотные правила, флаг.
- **Результат:**
  - unit-тест, воспроизводящий полную пилотную цепочку;
  - тест с выключенным флагом, доказывающий отсутствие регрессий;
  - тест с включённым флагом, доказывающий работу нового правила.
- **Критерий готовности:** все тесты проходят; тесты понятны и служат примером для будущих правил.
- **Зависимости:** Шаг 3.4.
- **Риски:**
  - Тесты завязаны на внутреннее устройство новой системы. *Как заметить:* при рефакторинге фазы 4 тесты приходится переписывать.
  - Недостаточно покрыты граничные случаи (шанс не сработал, цель иммунна). *Как заметить:* в игре правило ведёт себя иначе, чем в тесте.
- **Старая / новая система:** тесты покрывают обе системы; по умолчанию в игре работает старая, пилот включён только явно.

---

## Результат фазы

К концу фазы должны быть готовы:

1. Выбранный и документированный пилотный сценарий.
2. Работающий модификатор на интенте для одного случая.
3. Работающая контентная реакция на событие для одного случая.
4. Детерминированный рандом и лимит цепочки.
5. Набор тестов, покрывающих пилот с флагом и без.

---

## Точка проверки после фазы

- Игра запускается и проходит базовый сценарий с выключенным флагом пилота.
- С включённым флагом пилотного сценария срабатывает выбранное правило.
- Все guardian tests и новые пилотные тесты проходят.
- Нет дублирования эффектов вне пилотного сценария.
