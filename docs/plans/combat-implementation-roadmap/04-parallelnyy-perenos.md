# Фаза 4. Параллельный перенос

## Цель фазы

Последовательно перевести все контентные мировые реакции в data-driven правила, расширить `RuleContext` на новые события и подключить источники правил из экипировки, статусов и талантов. Старая система продолжает работать параллельно до фазы 5.

> **Примечание:** тайловые эффекты (`water`, `oil`, `fog`) и правила окружения вынесены из фазы 4 и из MVP. Они будут реализованы отдельно — после окончательного перехода на новый концепт боевой системы. См. фазу 7 «Тайловые эффекты и окружение» в [`README.md`](./README.md).

---

## Шаги

### ✅ Шаг 4.1. Перенос физики столкновений

> **Статус:** выполнен. Подробнее в [`phase-04/STATUS.md`](./phase-04/STATUS.md).

- **Вход:** классификация из фазы 1, каркас `ContentRuleReaction`, старые реакции `collisionDamageReaction` / `collisionStunReaction`.
- **Результат:**
  - правила `collision_damage` и `collision_daze` существуют в слое `world`;
  - при включённом флаге новой системы они срабатывают на `ENTITY_COLLIDED` + `displacement.push`;
  - поведение совпадает со старой системой.
- **Критерий готовности:** unit-тесты на столкновение с флагом проходят и дают тот же результат, что и без флага.
- **Зависимости:** Фаза 3.
- **Риски:**
  - Правило срабатывает при любом столкновении, а не только при насильственном толчке. *Как заметить:* обычное движение порождает `dazed`.
  - `collisionTargetId` разрешается неверно. *Как заметить:* урон/статус получает не та сущность.
- **Старая / новая система:** новые правила добавлены; старые реакции пока не отключены, поэтому флаг должен включать только одну из веток.

---

### ✅ Шаг 4.2. Перенос тиковых эффектов

> **Статус:** выполнен. Подробнее в [`phase-04/STATUS.md`](./phase-04/STATUS.md).

- **Вход:** старая реакция `burningTickReaction`, механизм тика статусов.
- **Результат:**
  - правило `burning_tick_damage` в слое `world`;
  - срабатывает на `STATUS_TICKED` + `burning`;
  - наносит урон согласно параметрам статуса.
- **Критерий готовности:** тест на горение совпадает со старой системой при включённом флаге.
- **Зависимости:** Шаг 4.1.
- **Риски:**
  - Тиковые события не содержат нужных тегов. *Как заметить:* правило не срабатывает.
  - Урон от тика дублируется с другими источниками. *Как заметить:* цель получает двойной урон.
- **Старая / новая система:** новое правило активно во флаге; старая реакция остаётся, но отключается в фазе 5.

---

### ✅ Шаг 4.3. Расширение `RuleContext` на остальные события

> **Статус:** выполнен. Подробнее в [`phase-04/STATUS.md`](./phase-04/STATUS.md).

- **Вход:** список событий, используемых переносимыми реакциями (`ENTITY_COLLIDED`, `STATUS_TICKED`, `TURN_BEGAN`, `ABILITY_USED`, `AP_RESTORED`, `ENTITY_HEALED` и др.).
- **Результат:** построитель `RuleContext` корректно обрабатывает все события, необходимые для MVP:
  - заполняет специфичные поля (`collisionTargetId`, `abilityTargetPosition`, `abilityTargets`, `eventDuration`, `eventStacks`);
  - корректно вычисляет `eventPosition` через fallback.
- **Критерий готовности:** для каждого нового события есть unit-тест на построение контекста.
- **Зависимости:** Шаги 4.1–4.2.
- **Риски:**
  - Неправильный контекст для `ABILITY_USED` при отсутствии цели. *Как заметить:* AoE-способности не попадают в правила.
  - `eventPosition` падает в `null` для событий без сущностей. *Как заметить:* мировые правила не срабатывают.
- **Старая / новая система:** обновлённый построитель используется новой системой; старая система не зависит от `RuleContext`.

---

### ✅ Шаг 4.4. Подключение источников правил: экипировка, статусы, таланты

> **Статус:** выполнен. Подробнее в [`phase-04/STATUS.md`](./phase-04/STATUS.md).

- **Вход:** жизненный цикл `activeRules` из фазы 2, шаблоны предметов/статусов/талантов.
- **Результат:**
  - при надевании предмета его `ruleIds` добавляются в `activeRules` актора;
  - при наложении статуса его `ruleIds` добавляются с уникальным `statusInstanceId`;
  - при получении таланта его `ruleIds` добавляются с учётом источника;
  - при снятии/истечении/отзыве правила удаляются точно.
- **Критерий готовности:** интеграционные тесты показывают, что кольцо, статус и талант активируют свои правила в бою.
- **Зависимости:** Шаг 4.3.
- **Риски:**
  - `activeRules` не обновляется при обновлении стека статуса. *Как заметить:* правила пропадают раньше времени или дублируются.
  - Два одинаковых предмета конфликтуют по `ownerContext`. *Как заметить:* снятие одного кольца снимает правила обоих.
- **Старая / новая система:** новые источники правил доступны новой системе; старая система не использует `activeRules`.

---

### Шаг 4.5. Модуль глобальных мировых контентных правил

- **Вход:** перенесённые реакции столкновений и тиков.
- **Результат:**
  - выделенный модуль/набор глобальных мировых правил;
  - они всегда активны в слое `world`;
  - порядок внутри слоя `world` зафиксирован для текущего MVP: глобальные мировые правила → (в будущем: тайловые эффекты → статические свойства тайла);
  - внутри подгруппы при равных `priority` используется tie-break по `ruleId`.
- **Критерий готовности:** тесты на порядок применения правил (например, сопротивление цели после мирового модификатора) проходят.
- **Зависимости:** Шаг 4.4.
- **Риски:**
  - Мировые правила жёстко привязаны к конкретному уровню/карте. *Как заметить:* поведение меняется при смене этажа.
  - Неправильный порядок внутри слоя `world`. *Как заметить:* статус тайла подавляет глобальную физику.
- **Старая / новая система:** новые мировые правила заменяют часть старых реакций; системные реакции остаются.

---

### Шаг 4.6. Поддержка мультитаргетных селекторов

- **Вход:** селекторы `allInRadius`, `nearestEnemy`, `chain`, каркас реакций.
- **Результат:**
  - правила с мультитаргетными селекторами корректно находят цели;
  - для каждой цели отдельно применяются `targetConditions`;
  - детерминированный выбор целей при равных условиях.
- **Критерий готовности:** unit-тесты на ауру, цепную молнию и ближайшего врага проходят.
- **Зависимости:** Шаг 4.4.
- **Риски:**
  - `allInRadius` включает владельца правила вне зависимости от фильтра. *Как заметить:* враждебная аура задевает самого владельца.
  - `chain` возвращается к уже выбранным целям. *Как заметить:* цепная молния бьёт одну и ту же цель дважды.
- **Старая / новая система:** новые возможности появляются только в новой системе; старая система их не поддерживает.

---

## Результат фазы

К концу фазы должны быть готовы:

1. Физика столкновений и тиковые эффекты перенесены в data-driven правила.
2. `RuleContext` поддерживает все события, нужные для MVP.
3. `activeRules` заполняется из экипировки, статусов и талантов.
4. Глобальные мировые контентные правила выделены и упорядочены.
5. Поддержаны мультитаргетные селекторы.

---

## Точка проверки после фазы

- Все guardian tests проходят с выключенным флагом новой системы.
- С включённым флагом поведение физических сценариев (столкновение, горение) совпадает с ожидаемым.
- Предметы, статусы и таланты активируют свои правила в бою.
- Игра запускается и проходит боевой сценарий без регрессий.
