# WP6.5 — Отчёт о проверке производительности content-rules

> **Статус:** выполнено. Baseline зафиксирован.
> **Ветка:** `concept/combat-system`
> **Дата:** 2026-07-16
> **Среда замера:** Windows, Node.js v26.1.0, Vitest 2.1.9

---

## 1. Цель и рамки

Цель рабочего пакета WP6.5 — понять, есть ли узкие места в декларативных content-rules на целевых боевых сценариях, и зафиксировать baseline для пост-MVP бэклога.

**Что измерялось:**

- Время выполнения типичной боевой цепочки (`GameSimulation.dispatch` + fireball + реакции).
- Время полного цикла на плотной карте (10 врагов, 3–5 active rules each, AoE).
- Время пересборки кэша `activeRules` для актора с большим числом правил.
- Производительность селекторов `allInRadius` / `nearestEnemy`.
- Накладные расходы подсистемы content-rules (`contentRulesEnabled=true/false`).

**Что не измерялось / вынесено в пост-MVP:**

- Накладные расходы события `RULE_TRIGGERED` — событие не реализовано в рамках текущей ветки (WP6.2 foundation). Оставлен TODO в perf-тесте и в разделе «Рекомендации».
- Браузерный рендеринг, анимации, UI — замер только симуляционного слоя.

---

## 2. Методика

- **Детерминизм:** фиксированный seed RNG (`SEED = 42`), программно созданная карта 20×20, видимость полностью открыта.
- **Изоляция:** глобальные мировые правила отключены в perf-тестах (`setWorldContentRulesOverride([])`), чтобы замер отражал только целевые реакции.
- **Тестовые правила:** реалистичные аналоги существующих правил (`fire_damage_ignites`, `counterattack_trigger`, `burning_tick_damage` и др.), но с детерминированным шансом 100%.
- **Статистика:** для каждого сценария собирались минимум, медиана, максимум, среднее и суммарное время. Итерации: 100 для боевых цепочек, 1000 для rebuild и селекторов.
- **Запуск:** `npm run test:perf` (отдельная конфигурация `vitest.perf.config.ts`). Обычный `npm test` perf-тесты не затрагивает.

### 3.2. Подробное описание сценариев

#### a. Типичная цепочка: fireball на 5 врагов (3 active rules each)

- **Карта:** 20×20.
- **Игрок:** в (5,5), способность `fireball`.
- **Враги:** 5 целей в радиусе AoE fireball (центр в (10,10)), у каждого 3 реакции: `perf_burning_on_damage`, `perf_poison_on_damage`, `perf_thorns`.
- **Действие:** `GameSimulation.dispatch({ type: 'USE_ABILITY', abilityId: 'fireball', targets: [(10,10)] })`.
- **Результат:** медиана **0.095 ms**, max 1.433 ms (выброс, вероятно, связан с GC). Далеко ниже целевого порога 16 ms.

#### b. Плотная карта: 10 врагов, 5 active rules each, AoE

- **Карта:** 20×20.
- **Враги:** 10 целей вокруг (10,10), у каждого 5 реакций: горение, яд, шипы, life drain, контратака.
- **Действие:** тот же fireball.
- **Результат:** медиана **0.091 ms**, max 0.404 ms. Рост числа врагов и rules не привёл к деградации в пределах замера.

#### c. Сбор activeRules: rebuild для актора с большим числом правил

- **Актор:** экипировка (3 предмета), 2 способности, 5 статусов с ruleIds; суммарно ~17 active rules.
- **Действие:** `rebuildActiveRules(actor)`.
- **Результат:** медиана **0.004 ms** за вызов. Пересборка кэша не является узким местом.

#### d. Селекторы: allInRadius / nearestEnemy

- **Карта:** 20×20, 100 врагов + игрок.
- **Радиус:** 5 (Chebyshev).
- **Результат:**
  - `allInRadius`: медиана **0.004 ms**.
  - `nearestEnemy`: медиана **0.003 ms**.
  - Высокий max у `allInRadius` (6.877 ms) — единичный выброс, не медиана.

#### e. Debug vs release: накладные расходы content-rules

- **Сценарий:** fireball на 5 врагов, сравнение `contentRulesEnabled=true` и `contentRulesEnabled=false`.
- **Результат:**
  - enabled: median **0.082 ms**.
  - disabled: median **0.066 ms**.
  - Разница в пределах погрешности: накладные расходы content-rules на измеренных цепочках незначительны.
- **TODO:** после реализации WP6.2 (событие `RULE_TRIGGERED`) заменить на прямой замер эмиссии/логирования RULE_TRIGGERED.

## 4. Анализ и выводы

### 4.1. Есть ли заметные фризы?

**Нет.** Все целевые сценарии показывают медиану **< 0.2 ms** на один intent/цепочку. Даже пиковые значения (кроме единичного GC-выброса в селекторе) остаются **< 1 ms**.

Целевой порог WP6.5 — *< 16 мс на intent на среднем ноутбуке разработчика* — выполнен с большим запасом.

### 4.2. Потенциальные узкие места

Хотя текущие цифры в норме, при дальнейшем росте контента возможны следующие узкие места:

1. **Линейный перебор `state.entities` в слое `radius`.**
   - `collectRules` для слоя `radius` проходит все сущности на карте при каждом событии.
   - При 100+ сущностях и частых событиях (каждый тик урона, каждое перемещение) это может стать заметным.
   - **Рекомендация:** пост-MVP внедрить пространственный индекс (например, сетка по чанкам) для наблюдателей в радиусе.

2. **Рост числа active rules на акторах.**
   - Сейчас 5 правил на врага дают пренебрежимый overhead.
   - При 20–30 правил на акторе и частых цепочках (например, каждый тик горения/яда) стоимость `collectRules` + `filterRulesByTrigger` + `evaluateConditions` вырастет линейно.
   - **Рекомендация:** ввести индекс activeRules по типу события/тегам, чтобы не сканировать весь кэш.

3. **Глубокие рекурсивные цепочки реакций.**
   - Лимит 1000 предотвращает бесконечные циклы, но длинные цепочки (огонь → горение → тик → урон → реакция → ...) могут вызывать много итераций `executeIntent`.
   - **Рекомендация:** пост-MVP добавить метрики глубины цепочек и журналирование «горячих» правил.

4. **RULE_TRIGGERED (не реализовано).**
   - После WP6.2 каждая сработавшая реакция будет порождать событие `RULE_TRIGGERED`, которое пишется в дерево `ExecutionNode`.
   - В debug-режиме это может увеличить расход памяти и размер дерева выполнения.
   - **Рекомендация:** после foundation WP6.2 перезапустить `test:perf` и сравнить результаты.

### 4.3. Baseline-рекомендации

- **Принять текущие цифры как baseline MVP:**
  - typical chain: **~0.10 ms median**.
  - dense map: **~0.09 ms median**.
  - rebuild activeRules: **~0.003 ms**.
  - selectors: **~0.003–0.004 ms**.
- **Регрессионный барьер:** не допускать медианы typical chain > 1 ms и dense map > 2 ms без явной оптимизации.
- **CI:** perf-тесты запускаются отдельно (`npm run test:perf`) и не замедляют обычный `npm test`.
