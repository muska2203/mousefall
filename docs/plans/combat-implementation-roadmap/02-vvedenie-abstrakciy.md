# Фаза 2. Введение абстракций

## Цель фазы

Создать каркас новой системы правил рядом с существующей, но **ещё не подключать её к боевому циклу**. Все новые механизмы должны быть покрыты unit-тестами в изоляции и управляться через флаг/переключатель.

---

## Шаги

### Шаг 2.1. Реестр декларативных правил

- **Вход:** контракты из фазы 1 и решение о формате хранения правил.
- **Результат:**
  - отдельное место для хранения правил;
  - возможность находить правило по `ruleId`;
  - базовая валидация: все `ruleIds`, на которые ссылаются шаблоны, существуют, в одном шаблоне нет дублей.
- **Критерий готовности:** unit-тесты успешно загружают реестр, находят известные правила и отлавливают ссылки на несуществующие `ruleIds`.
- **Зависимости:** Фаза 1.
- **Риски:**
  - Раздувание реестра одноразовыми правилами. *Как заметить:* правила не переиспользуются между предметами/статусами.
  - Циклические зависимости между реестром и контентом. *Как заметить:* нарушение правил слоёв из [`docs/agents/LAYERS.md`](../../agents/LAYERS.md).
- **Старая / новая система:** старая система не использует реестр; новый реестр существует, но не влияет на игру.

---

### Шаг 2.2. Построитель контекста правил (`RuleContext`)

- **Вход:** согласованный контракт `RuleContext` из фазы 1, список событий, которые покрываются в MVP.
- **Результат:** функция/модуль, который по событию или интенту строит `RuleContext`, включая `sourceEntityId`, `targetEntityId`, `eventPosition`, `eventTags` и специфичные поля.
- **Критерий готовности:** для каждого поддерживаемого события (`ENTITY_DAMAGED`, `ENTITY_HEALED`, `ENTITY_COLLIDED`, `STATUS_APPLIED`, `ABILITY_USED`, `TURN_BEGAN`, `AP_RESTORED` и др.) есть unit-тест, проверяющий корректность построенного контекста.
- **Зависимости:** Шаг 2.1.
- **Риски:**
  - Неправильный `eventPosition` при отсутствии явной позиции в событии. *Как заметить:* тесты с `AP_RESTORED` или `RESOURCE_CONSUMED` дают `null` вместо позиции сущности.
  - Особые поля (например, `collisionTargetId`) заполняются неверно. *Как заметить:* правила столкновений срабатывают не на том объекте.
- **Старая / новая система:** новый построитель не вызывается в боевом цикле; старый поток не изменён.

---

### Шаг 2.3. Жизненный цикл `activeRules`

- **Вход:** контракты `ActiveRule` и `OwnerContext`, механизмы экипировки, статусов и талантов.
- **Результат:**
  - у актора появляется отдельное производное поле `activeRules`, не совмещённое со списком активных статусов;
  - реализованы операции добавления/удаления правил при:
    - надевании и снятии предмета,
    - наложении и снятии статуса,
    - получении и отзыве таланта;
  - `ownerContext` корректно заполняется для каждого источника (`itemInstanceId`, `statusInstanceId`, `abilityId:sourceItemInstanceId`).
- **Критерий готовности:** unit-тесты показывают, что `activeRules` обновляется в точности при изменении источников и не содержит лишних записей.
- **Зависимости:** Шаг 2.1.
- **Риски:**
  - При обновлении длительности статуса правила удаляются/добавляются лишний раз. *Как заметить:* `statusInstanceId` нестабилен.
  - Один и тот же талант из разных предметов конфликтует. *Как заметить:* отзыв таланта удаляет правила от обоих источников.
- **Старая / новая система:** новый кэш существует, но не используется боевым циклом; старая логика экипировки/статусов/талантов работает как раньше.

---

### Шаг 2.4. Каркас `ContentRuleReaction`

- **Вход:** реестр правил, построитель `RuleContext`, `activeRules`.
- **Результат:**
  - функция, которая по `state` и `RuleContext` собирает активные правила из слоёв `source`, `target`, `world`, `radius`;
  - фильтрует их по событию и тегам;
  - применяет условия и селекторы целей;
  - возвращает список интентов, **но пока не исполняет их**.
- **Критерий готовности:** unit-тесты проверяют, что простые правила (`chance`, `hasStatus`, селекторы `eventTarget`/`self`) порождают ожидаемые интенты для фиктивного состояния.
- **Зависимости:** Шаги 2.1–2.3.
- **Риски:**
  - Неправильный порядок слоёв. *Как заметить:* правила цели применяются раньше правил источника.
  - Дублирование правил при `sourceEntityId === targetEntityId`. *Как заметить:* самобафы срабатывают дважды.
- **Старая / новая система:** каркас существует, но не вызывается из обработки событий; старые мировые реакции остаются единственными.

---

### Шаг 2.5. Каркас модификаторов на интенте

- **Вход:** реестр правил, `activeRules`, поддержка событий-интентов (`DAMAGE_INTENT` и др.).
- **Результат:**
  - функция, которая получает интент и `RuleContext`;
  - собирает модифицирующие правила из слоёв `source → target → world → radius`;
  - применяет множители и добавки в правильном порядке внутри слоя;
  - возвращает изменённый интент, **не мутируя состояние мира**.
- **Критерий готовности:** unit-тесты показывают, что `MODIFY_DAMAGE` с `multiply` и `add` изменяет урон интента предсказуемо; добавленные теги присутствуют.
- **Зависимости:** Шаги 2.1–2.3.
- **Риски:**
  - Модификаторы мутируют `state`. *Как заметить:* тесты фиксируют изменения до вызова `IntentExecutor`.
  - Порядок «сначала multiply, потом add» нарушается при переходе между слоями. *Как заметить:* результат урона не совпадает с ожидаемым.
- **Старая / новая система:** новый модификатор не вызывается из боевого цикла; старая логика урона работает без изменений.

---

### Шаг 2.6. Флаги и точки включения

- **Вход:** каркасы из шагов 2.4 и 2.5.
- **Результат:**
  - чётко выделенные точки, в которые в будущем встраиваются модификатор и `ContentRuleReaction`;
  - механизм включения/выключения новой системы (feature flag, отдельная ветка, параметр конфигурации);
  - при выключенном флаге боевой цикл работает исключительно на старой системе.
- **Критерий готовности:** при выключенном флаге все guardian tests проходят; при включённом флаге (пока без реальных правил) не появляется нового поведения.
- **Зависимости:** Шаги 2.4 и 2.5.
- **Риски:**
  - Флаг рассасывается по коду. *Как заметить:* невозможно быстро отключить новую систему.
  - Точки включения меняют старый поток даже при выключенном флаге. *Как заметить:* guardian tests падают.
- **Старая / новая система:** флаг отключает новую систему; старая система остаётся рабочей по умолчанию.

---

## Результат фазы

К концу фазы должны быть готовы:

1. Реестр декларативных правил с базовой валидацией.
2. Построитель `RuleContext` для поддерживаемых событий.
3. Производный кэш `activeRules` с жизненным циклом.
4. Изолированный каркас `ContentRuleReaction`.
5. Изолированный каркас модификаторов на интенте.
6. Механизм флагов и зафиксированные точки включения.

---

## Точка проверки после фазы

- Все unit-тесты новых модулей проходят.
- Все guardian tests проходят с выключенным флагом новой системы.
- Игра запускается и играется без видимых изменений.
- Новая система присутствует в коде, но не влияет на боевой цикл.
