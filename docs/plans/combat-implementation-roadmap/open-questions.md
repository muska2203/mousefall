## 1. Интеграция с текущим потоком Intent → Event

### Вопрос
Как именно новый слой модификаторов должен быть встроен в существующий `executeIntent`?

### Решение ✅
Использовать **Вариант А**: центральная точка применения модификаторов внутри общей функции `executeIntent` перед вызовом конкретного `IntentExecutor`. Неподдерживаемые в MVP интенты проходят прозрачно. Включение управляется runtime-флагом в `Simulation`; в пилоте флаг активируется только для выбранного события.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 2. Источник интента

### Вопрос
Достаточно ли текущего `Intent` для однозначного определения `sourceEntityId`/`targetEntityId` и тегов, или требуется расширить/уточнить его контракт?

### Решение ✅
Использовать **Вариант А**: не менять union `Intent` глобально. Для пилота `RuleContext` builder читает имеющиеся поля у нужных интентов (в первую очередь `DAMAGE_INTENT`). Глобальная нормализация `Intent` отложена на пост-MVP: добавлена в [`06-polirovka.md`](./06-polirovka.md) / бэклог.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 3. Пилот: с чего начать

### Вопрос
Какой сценарий выбрать для первой реальной проверки?

### Решение ✅
Пилот идёт в два шага: сначала **реакция «огненный урон → горение»** (любой огненный урон имеет шанс поджечь цель), затем сразу **модификатор «огненный урон ×1,5»**. Presentation/UI в пилоте не трогаются. Для модификатора огня временно изолируется старая `fireDamageReaction`, чтобы избежать дублирования.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 4. Текущее состояние `activeRules`

### Вопрос
Есть ли в коде уже что-то похожее на кэш активных правил у актора, или его нужно создавать с нуля?

### Решение ✅
У актора есть список активных статусов, но это другое. `activeRules` создаётся как отдельное поле и ведётся с нуля по контракту концепта.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 5. Тестовое покрытие боевых сценариев

### Вопрос
Какие существующие тесты считать обязательными для проверки после каждого шага?

### Решение ✅
Существующие тесты остаются базовым guardian-набором. Детальные тесты на переносимые сценарии добавляются непосредственно в той фазе, где функциональность переносится в новую систему, а не заранее всей пачкой.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 6. Формат хранения правил

### Вопрос
Устраивает ли подход, при котором сами правила — TypeScript-объекты в `src/simulation/content-rules/`, а шаблоны ссылаются по `ruleIds`?

### Решение ✅
Оставляем как в концепте: правила — TypeScript-объекты, шаблоны ссылаются по `ruleIds`. JSON-вариант не рассматривается для MVP.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 7. Представление новых событий в UI

### Вопрос
Могут ли новые события (`STATUS_BLOCKED`, `TILE_EFFECT_CHANGED`, `EXPLOSION_TRIGGERED`) игнорироваться UI на первых порах, или для них сразу нужна визуализация?

### Решение ✅
В пилоте Presentation/UI не трогаются. После готовности симуляционного слоя (фазы 1–5) добавляется отдельная фаза адаптации Presentation согласно [`docs/plans/Презентационный_слой_и_UI.md`](../../plans/Презентационный_слой_и_UI.md). Только после этого новые правила добавляются сразу в симуляцию и Presentation/UI.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

## 9. Порядок мировых правил внутри слоя `world`

### Вопрос
Нужно ли на MVP фиксировать точный порядок «глобальные мировые правила → тайловые эффекты → статические свойства тайла» или достаточно общего принципа?

### Решение ✅
На MVP фиксируем точный порядок: глобальные мировые правила → тайловые эффекты → статические свойства тайла. Внутри подгруппы при равных `priority` используется tie-break по `ruleId`. Порядок прописывается в плане и decision log.

Подробное обоснование зафиксировано в [`decision-log.md`](./decision-log.md).

### Почему важно
Влияет на предсказуемость физических комбо (например, столкновение + лужа масла).
