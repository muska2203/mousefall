# Фаза 5.5. Адаптация Presentation

> Статус: выполнено.

## Цель фазы

После того как симуляционный слой полностью перешёл на новую систему правил, подготовить Presentation Layer к отображению новых событий и эффектов. После этой фазы любое новое правило будет добавляться сразу и в симуляцию, и в Presentation/UI.

---

## Шаги

### Шаг 5.5.1. Инвентаризация новых событий для Presentation

- **Вход:** дерево `ExecutionNode`, полученное в результате работы новой системы правил.
- **Результат:**
  - список событий, которые появляются благодаря новым правилам (`STATUS_BLOCKED` и др.); события тайловых эффектов (`TILE_EFFECT_CHANGED`, `EXPLOSION_TRIGGERED`) отложены на фазу 7.
  - для каждого события зафиксировано, нужно ли обновлять `DisplayState`, добавлять строку в combat log, проигрывать анимацию.
- **Критерий готовности:** список согласован; нет событий, которые теряются на границе Simulation → Presentation.
- **Зависимости:** Фаза 5.
- **Риски:**
  - Какое-то событие не замечено. *Как заметить:* в игре виден финальный результат, но не промежуточные эффекты.
  - Событие дублируется в log. *Как заметить:* одно действие порождает несколько одинаковых строк.
- **Старая / новая система:** новая система порождает события; Presentation ещё не обрабатывает их.

---

### Шаг 5.5.2. Обновление `DisplayState` для новых событий

- **Вход:** список событий из шага 5.5.1, `DisplayState`, [`Презентационный_слой_и_UI.md`](Презентационный_слой_и_UI.md).
- **Результат:**
  - Presentation корректно применяет патчи от новых событий;
  - состояние сущностей и статусов отражает результат цепочки правил;
  - добавлен generic fallback-ресинк для неизвестных событий, если это необходимо для консистентности.
- **Критерий готовности:** после боевого действия `DisplayState` совпадает с `GameState` по статусам (тайловые эффекты — фаза 7).
- **Зависимости:** Шаг 5.5.1.
- **Риски:**
  - Патч применяется дважды. *Как заметить:* статус появляется и исчезает в течение одной анимации.
  - Ресинк перезаписывает промежуточные состояния, нужные для анимации. *Как заметить:* анимация прыгает.
- **Старая / новая система:** Presentation учится понимать события от новой системы; старые события обрабатываются как раньше.

---

### Шаг 5.5.3. Добавление строк combat log

- **Вход:** список новых событий, [`Презентационный_слой_и_UI.md`](Презентационный_слой_и_UI.md).
- **Результат:**
  - для значимых событий появляются понятные строки в combat log;
  - лог не захламлён второстепенными событиями;
  - строки локализуемы через существующую систему i18n.
- **Критерий готовности:** тестовая цепочка с огнём/горением отображается в log коротко и понятно.
- **Зависимости:** Шаг 5.5.2.
- **Риски:**
  - Слишком много строк на одно действие. *Как заметить:* log переполняется.
  - Недостаточно информации. *Как заметить:* игрок не понимает, почему цель загорелась.
- **Старая / новая система:** новые события получают строки log; старые события сохраняют свои строки.

---

### Шаг 5.5.4. Добавление анимаций для новых эффектов

- **Вход:** список новых визуальных эффектов (поджог, блокировка статуса); взрывы и изменения тайловых эффектов — фаза 7.
- **Результат:**
  - для каждого значимого события определена анимационная команда;
  - анимации не блокируют ввод и не ломают последовательность ходов;
  - где нет готового ассета, используется временная визуализация.
- **Критерий готовности:** пилотная цепочка с огнём/горением проигрывается в UI последовательно и понятно.
- **Зависимости:** Шаг 5.5.3.
- **Риски:**
  - Анимация нового эффекта конфликтует с существующей. *Как заметить:* всплывающий урон и анимация горения накладываются друг на друга.
  - Отсутствие ассета блокирует фазу. *Как заметить:* временная визуализация не была предусмотрена.
- **Старая / новая система:** новые эффекты получают анимации; старые анимации не меняются.

---

### Шаг 5.5.5. Тестирование Presentation

- **Вход:** фикстуры и деревья событий от новой системы правил.
- **Результат:**
  - unit-тесты Presentation проверяют перевод новых событий в `DisplayState`, log и анимации;
  - интеграционные тесты проходят полную цепочку: действие → Simulation → Presentation → ViewModel.
- **Критерий готовности:** тесты проходят; UI корректно отображает пилотную цепочку.
- **Зависимости:** Шаг 5.5.4.
- **Риски:**
  - Тесты Presentation зависят от внутреннего устройства анимаций. *Как заметить:* при смене renderer тесты ломаются.
  - Покрыты только happy path. *Как заметить:* граничные случаи (шанс не сработал, статус заблокирован) не отображаются.
- **Старая / новая система:** тесты покрывают новые события от новой системы; старые тесты Presentation остаются.

---

### Шаг 5.5.6. Фиксация контракта «новое правило = симуляция + Presentation»

- **Вход:** результаты шагов 5.5.1–5.5.5.
- **Результат:**
  - documented соглашение: любое новое правило после этой фазы должно включать как логику в Simulation, так и представление в Presentation/UI;
  - чеклист добавления нового правила обновлён.
- **Критерий готовности:** в `docs/agents/` или `docs/plans/` зафиксирована инструкция, а checklist нового правила включает пункт про Presentation.
- **Зависимости:** Шаг 5.5.5.
- **Риски:**
  - Контракт не соблюдается. *Как заметить:* новое правило добавлено, но в UI оно не видно.
- **Старая / новая система:** новое правило теперь разрабатывается для обоих слоёв; старые правила не трогаются.

---

## Результат фазы

К концу фазы должны быть готовы:

1. Список новых событий, требующих обработки в Presentation.
2. Корректное обновление `DisplayState` для этих событий.
3. Строки combat log и анимации для ключевых эффектов.
4. Тесты Presentation на новые события.
5. Зафиксированный контракт, что новые правила добавляются в оба слоя.

## Результат выполнения

Фаза реализована:

- Новая модель `DisplayState`, патчи `DisplayPatch` и узлы `PresentationNode` — `src/presentation/displayState/`.
- Animation builders для новых событий — `src/presentation/animation/builders/`.
- Регистрация builders — `src/presentation/animation/register.ts`.
- Обновлённый combat log — `src/presentation/logBuilder.ts`.
- Локализации log — `src/i18n/schema.ts`, `src/i18n/locales/ru/system/logBuilder.ts`, `src/i18n/locales/en/system/logBuilder.ts`.
- Контракт задокументирован — `docs/agents/PRESENTATION_CONTRACT.md`.
- Чеклист добавления Event обновлён — `docs/agents/ACTION_SYSTEM.md`.

> **Примечание:** события тайловых эффектов (`TILE_EFFECT_CHANGED`, `EXPLOSION_TRIGGERED`) отложены на фазу 7 и в контракт 05.5 не входят.

## Точка проверки после фазы

- Игра проходит пилотную цепочку с огнём/горением и корректно отображает её в UI.
- Combat log содержит ключевые события, но не захламлён.
- Анимации проигрываются последовательно и не ломают ход.
- Все тесты Simulation и Presentation проходят.
